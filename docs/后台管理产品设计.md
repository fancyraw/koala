# 后台管理产品设计

## 一、概述

后台供管理员进行商城运营管理，覆盖商品、营销、订单、用户、内容、系统配置与运维等模块。

> C 端产品需求见《C端产品设计.md》；UI 设计见《后台管理UI设计.md》、《后台管理UI原型.html》。

## 二、角色与权限

| 角色 | 可访问范围 |
|------|-----------|
| 普通管理员 admin | 数据看板、商品、订单、用户、优惠券、营销配置、内容、系统设置 |
| 超级管理员 super_admin | 以上全部 + **运维管理**（系统状态 / 操作日志 / 数据操作） |

- 所有写操作记入操作日志（操作人 + 时间 + 类型 + 内容）
- 禁用用户后该用户无法登录商城

## 三、模块清单

| 模块 | 主要职责 |
|------|---------|
| 数据看板 | 核心经营指标与趋势分析（含退款、优惠券核销数据） |
| 商品管理 | 商品列表（含编辑表单）、分类管理（工具栏入口）、库存——**单级菜单** |
| 订单管理 | 订单查询、发货、物流单号、退款 / 售后处理 |
| 用户管理 | 用户列表、会员等级、用户详情 |
| 优惠券管理 | 券模版的创建 / 编辑 / 停发、核销数据、**券发放明细**（用户券由系统下发） |
| 营销配置 | 会员等级**可视化编辑**（门槛走 sys_config、礼券绑定走 member_level_gift） |
| 内容管理 | Banner |
| 系统设置 | 配送（运费 / 包邮门槛）、支付配置、订单超时配置 |
| 运维管理 | 系统状态、操作日志、数据操作（仅超级管理员） |

> **说明**：本商城**不设积分体系**；会员权益为「升级时一次性发放专属优惠券」（升级送券），无下单折扣、无积分。

## 四、各模块产品需求

### 1. 数据看板

- **入口**：登录后默认首页 / 侧栏「数据看板」
- **时间段切换**：今日 / 本周 / 本月，影响全部指标与图表
- **核心指标卡**（4 张）：今日订单、今日销售额、新增用户、总用户数；含环比（↑绿 / ↓红）
- **第二行指标卡**（4 张，补充运营数据）：
  - 待处理退款：当前待审核的退款申请数量，红色高亮提醒
  - 今日退款额：当日已退款总金额
  - 优惠券核销率：进行中优惠券的整体核销率（已用 / 已领）
  - 今日客单价：当日销售额 / 当日订单数
- **分析 Tab**：
  - 销售趋势：折线图（日期 × 销售额）
  - 热销商品：排行表（商品 / 销量 / 销售额），「查看全部」跳商品管理
  - 订单分布：各状态订单占比饼图
  - 用户分析：新增用户趋势 + 会员等级分布
  - 退款分析：退款趋势（日期 × 退款额）+ 退款原因分布饼图
- 刷新按钮重载全部数据

### 2. 商品管理

> **导航说明**：商品管理为**单级菜单**（不再有「商品列表 / 分类管理」二级子菜单）。分类管理入口收进商品列表页工具栏，作为「分类管理」按钮放在「+ 新增商品」旁边——分类是商品的属性维度，与商品同屏管理更顺手，也省去一层菜单跳转。

#### 2.1 商品列表
- **工具栏**：搜索（商品名）+ 分类筛选 + 状态筛选（上架 / 下架 / 售罄）；右侧操作区「分类管理」次按钮 +「+ 新增商品」主按钮

#### 2.1.1 商品编辑（新增 / 编辑表单）

```
┌──────────────────────────────────────────────────────┐
│ ‹ 返回商品列表          新增商品 / 编辑商品              │
├──────────────────────────────────────────────────────┤
│ 商品名称  [____________________________]                │
│ 商品主图  [＋上传]  （1 张，建议 1:1）                   │
│ 商品分类  [蜂蜜 ▾]                                       │
│ 商品标签  ○无  ○热销  ○新品  ○礼盒  ○特惠（单选，可不选）│
│ 是否推荐  [○]  （开启进入 C 端首页精选推荐区）            │
│ ── 规格与价格（最多 4 个）──                             │
│  规格名 [500g]  售价 [¥88]  库存 [150]   [删除]         │
│  规格名 [1kg ]  售价 [¥158] 库存 [80 ]   [删除]         │
│                                          [+ 添加规格]   │
│ ── 产品亮点（多条，首条复用为 C 端推荐种草语）──          │
│  · [纯天然零添加_________________]  [删除]              │
│  · [古法熬制工艺_________________]  [删除]              │
│                                          [+ 添加亮点]   │
│ 详情图    [＋上传多张]  （长图 / 多图，C 端详情页展示）   │
│                                       [取消] [保存]     │
└──────────────────────────────────────────────────────┘
```

- **基础字段**：商品名称、主图（1 张）、分类（下拉）、标签（单选，可不选）、是否推荐（开关）
- **规格与价格**：每条含 规格名 / 售价 / 库存；最多 4 个；单规格商品填一条即可（C 端规格区不展示）
- **产品亮点**：多条文本，**首条自动复用为 C 端精选推荐卡的种草语**（无需单独录入推荐语）；无亮点则 C 端不展示该行
- **详情图**：可上传多张，C 端商品详情页展示
- **无营销开关**：表单内不含任何营销 / 折扣开关，所有商品默认统一参与满减券与无门槛券
- **保存校验**：名称、主图、分类、至少 1 条规格（含售价、库存）为必填；保存成功 Toast 后返回列表
- **列表字段**：主图、名称、售价、标签、是否推荐、库存、分类、状态、操作
- **库存预警**：库存 < 20 显示黄色「库存不足」；库存 = 0 显示红色「已售罄」并自动置为售罄状态
- **操作**：新增 / 编辑 / 上架 / 下架；分页
- **商品字段**：名称、主图、售价（只售价无划线价）、规格（最多 4 个，可对应不同价格）、标签（热销 / 新品 / 礼盒 / 特惠，**单选，可不选**；用于 C 端商品卡片左上角角标，每个商品最多展示一个）、**是否推荐（用于 C 端首页精选推荐区）**、库存、分类、详情图、产品亮点
- **营销参与（统一规则，无单品开关）**：所有商品默认参与满减券与无门槛券，无需逐个配置开关；C 端的券标签、到手价计算对全部商品统一生效。**会员无下单折扣，亦无任何商品营销开关**

#### 2.2 分类管理
- **入口**：商品列表页工具栏「分类管理」按钮 → 打开分类管理弹窗 / 抽屉（不占独立菜单）
- **字段**：分类名称、图标、排序、状态
- **操作**：新增 / 编辑 / 启用 / 禁用 / 删除；拖拽 `≡` 调整排序
- **规则**：
  - 同步到 C 端首页品类专区与商品列表筛选栏
  - **仅商品数量为 0 的分类可删除**
  - 禁用的分类 C 端不展示

### 3. 订单管理

```
┌──────────────────────────────────────────────────────┐
│ 订单管理                                      [导出订单] │
├──────────────────────────────────────────────────────┤
│ [全部][待付款][待发货][待收货][已完成][已取消][退款售后]  │
│ [🔍订单号/买家姓名]                                     │
├──────────────────────────────────────────────────────┤
│ 订单号│买家│商品│金额│状态│下单时间│操作                │
│ ..01 │张三│百花蜜+2│¥78│待发货│06-10│ 详情  发货      │
│ ..98 │李四│蜂王浆 │¥168│退款售后│06-09│详情 处理退款   │
│  ▼ 展开行：收货人/地址/商品明细/优惠券/运费/实付/备注    │
└──────────────────────────────────────────────────────┘

  ┌── 发货弹窗（仅待发货）──┐   ┌── 处理退款弹窗（仅退款售后）──┐
  │ 物流公司 [顺丰 ▾]      │   │ 退款类型：仅退款 / 退货退款    │
  │ 物流单号 [SF________] │   │ 退款金额：¥168（红色大字）     │
  │        [取消][确认发货]│   │ 退款原因 / 补充说明 / 凭证图   │
  └──────────────────────┘   │      [拒绝(填理由)] [同意]    │
                             └──────────────────────────────┘
```

- **操作列按状态变化**：
  - 待付款 → 仅「详情」（不可代付）
  - 待发货 → 「详情」+「发货」
  - 待收货 / 已完成 → 「详情」（已完成可在详情内手动发起退款）
  - 退款售后 → 「详情」+「处理退款」（退货退款同意后变为「确认收货」）
- **状态 Tab**（附数量）：全部 / 待付款 / 待发货 / 待收货 / 已完成 / 已取消 / 退款售后
- **搜索**：订单号 / 买家姓名
- **列表**：订单号、买家、商品（首件 + 共 N 件）、金额、状态、下单时间、操作；点击行展开详情（收货人 / 地址 / 商品明细 / 优惠券 / 运费 / 实付 / **订单备注**）
- **导出订单**：导出当前筛选条件下的订单
- **发货**：仅待发货订单可操作，弹窗填写**物流公司 + 物流单号** → 转待收货；单号同步 C 端订单详情物流时间轴
- **退款 / 售后处理**（核心，防货财两空）：
  - **仅退款（未发货）**：待发货订单申请退款，商家审核通过后**直接原路退款**，无需寄回
  - **退货退款（已发货 / 已收货）**：流程「**提交申请 → 商家审核 → 买家寄回 → 商家确认收货 → 退款**」，**商家确认收货后才打款**
  - 「处理退款」支持：同意 / 拒绝、填写退款金额、记录买家寄回单号、确认收货并打款
  - 退款资金与权益：现金原路退回；优惠券若未过期原路退回
- **超时取消**：待付款订单超过 30 分钟未支付自动取消（超时时长可在系统设置中配置）
- **售后处理详细流程**：
  - 列表中退款售后订单展示**售后状态子标签**：待审核 / 已同意·待寄回 / 已寄回·待确认 / 已退款 / 已拒绝
  - **审核操作**：点击「处理退款」弹窗，展示买家申请信息（退款原因 / 退款金额 / 补充说明 / 凭证图片），操作「同意」/「拒绝」（拒绝需填写理由）
  - **仅退款（未发货）**：同意后系统自动原路退款，订单转「已退款」
  - **退货退款（已发货）**：同意后等待买家寄回 → 买家填写寄回单号后订单展示寄回物流信息 → 商家点击「确认收货并退款」→ 系统原路退款
  - **退款资金与权益处理**：现金原路退回；未过期优惠券原路退回
  - **已完成订单售后**：C 端引导至客服，后台可在订单详情中手动发起退款（需填写退款金额与原因）

### 4. 用户管理

- **搜索**：昵称 / 手机号
- **列表**：头像、昵称、手机号、**会员等级**、累计消费、订单数、注册时间、状态、操作
- **会员等级**：由累计消费自动判定——🥉 普通（注册即享）/ 🥈 白银（≥500 元）/ 🥇 黄金（≥1500 元）。**会员权益为升级送券**（无折扣、无积分），升级门槛与礼券可在「营销配置」中调整
- **用户详情页**（点击「详情」进入）：
  - **基本信息**：头像、昵称、手机号、会员等级、累计消费、注册时间、状态
  - **消费记录 Tab**：该用户的订单列表（订单号 / 商品 / 金额 / 状态 / 时间），可点击查看订单详情
  - **优惠券 Tab**：该用户持有的优惠券列表（券名 / 类型 / 来源：新人 / 升级礼券 / 活动 / 状态：可用·已使用·已过期 / 使用时间）
- **禁用**：切换状态，禁用后无法登录商城
- **新人定义**：注册 / 登录后**且无历史已支付订单**的用户（用于新人券发放资格校验）

### 5. 优惠券管理

> **核心模型：券模版（coupon）+ 用户券（user_coupon）。** 后台只创建并管理「券模版」（定义券的类型、面额、有效期、总量等规则）；用户实际持有的是由系统**下发**生成的「用户券」实例。**全程系统下发、无手动领取**——用户在登录 / 进首页时由前端查到新到账券，弹窗提示「已为你发放 N 张券」，按钮「去使用」（详见《C端产品设计.md》优惠券章节）。这样可避免「领券不下单」的羊毛账户占用名额与库存。

- **状态 Tab**：全部 / 进行中 / 未开始 / 已结束 / 已停发
- **券卡展示**（券模版维度）：券名、类型、有效期、限领、**已发 X/总量**、**已用 + 核销率**、状态、操作（编辑 / 停发 / 删除 / **发放明细**）
- **新建 / 编辑字段（券模版）**：券名、券类型（满减 / 无门槛）、优惠金额（满减券额外填门槛金额）、有效期、每人限领张数、发行总量
- **券发放明细（券 → 用户的下发关系）**：券卡上「发放明细 ›」打开弹窗 / 抽屉，列出该券模版下发的全部用户券——列 **用户（昵称 / 手机号）/ 状态（未使用 / 已使用 / 已过期）/ 下发时间 / 使用时间**，支持按状态筛选 + 分页。与「用户详情页 → 优惠券 Tab」（用户 → 券）互为反向视角，便于核查某张券发给了谁、用了多少（数据取自 user_coupon 按 coupon_id 查，无需新增表）
- **下发场景（系统按规则自动生成用户券，无手动领取）**：
  1. **新人券**：用户满足「新人」条件（无历史已支付订单）时由系统下发
  2. **升级礼券**：累计消费达门槛升级时按「营销配置」绑定的券模版下发（见 6.1）
  3. **活动券**：后台按活动发放条件下发
- **核销数据**：已发数量 = 下发的用户券总数；已用数量 = 状态为「已使用」的用户券数；核销率 = 已用 / 已发
- **规则**：
  - **已开始（已有下发记录）的券模版不可删除，只能停发**；停发后不再生成新的用户券，已发出的用户券不受影响
  - 券模版状态：未开始 / 进行中 / 已结束 / 已停发
- **券组合默认始终最优**：下单时系统自动从用户可用券中选「无门槛 + 满减各一张」抵扣最大的组合，无开关、无需配置

### 6. 营销配置

> **设计取舍**：会员升级门槛是标量、升级礼券绑定走 `member_level_gift` 关系表（等级 × 券模版 × 张数），两者都结构化、可校验，因此**直接在后台可视化编辑**——门槛改 sys_config、礼券绑定增删改 member_level_gift。升级礼券通过**绑定券模版**下发，不在此内联配置面额，避免重复定义。

#### 6.1 会员等级与升级礼券（可编辑）
- **等级列表**（门槛与礼券绑定均可在后台编辑）：

  | 等级 | 升级门槛（累计消费） | 升级礼券（绑定券模版） | 当前人数 | 操作 |
  |------|---------------------|----------------------|---------|------|
  | 🥉 普通会员 | 注册即享（固定） | 无（注册即享，不可配） | 180 | — |
  | 🥈 白银会员 | ≥ 500 元（可改） | 新人立减5 ×1 + 满200减20 ×1 | 86 | 编辑 |
  | 🥇 黄金会员 | ≥ 1500 元（可改） | 立减10 ×2 + 满300减50 ×1 | 20 | 编辑 |

- **编辑交互**：
  - **升级门槛**：点击门槛数字可直接编辑（输入框 + 元），保存写入 `sys_config:member.silver_threshold` / `gold_threshold`
  - **升级礼券绑定**：每个等级一个礼券绑定区，可**增删行**——每行「选择券模版（下拉，取自优惠券管理）+ 张数」；保存写入 `member_level_gift` 表（校验 coupon 外键）。普通会员无礼券绑定区
  - 底部「保存」主按钮 + 成功 Toast
- **配置存储**：
  - **升级门槛金额**：sys_config（`member.silver_threshold` / `member.gold_threshold`，标量）
  - **升级礼券绑定**：`member_level_gift` 关系表（等级 × 券模版 ID × 张数），复用「优惠券管理」的券模版，避免重复定义、可关联查询、外键校验
- **规则**：
  - 等级数量固定 3 级；普通会员无礼券（注册即享）
  - 门槛改动后实时生效，已有用户按新门槛重新判定等级
  - 礼券绑定只能选**已存在的券模版**；停发 / 删除的券模版不可被新绑定
  - **会员无下单折扣、无积分**；权益完全体现为升级时一次性下发的专属优惠券（绑定的券模版）
  - **升级触发**：累计消费（以支付成功订单累计，退款扣回）达门槛时自动升级并按 member_level_gift 下发对应券模版生成的用户券；同一等级礼券只发放一次
- **会员统计**：各等级用户数量分布（普通 N 人 / 白银 N 人 / 黄金 N 人），实时统计

### 7. 内容管理

#### 7.1 Banner
- **字段**：图片、跳转链接、排序、上下线状态
- **操作**：新增 / 编辑 / 上线 / 下线 / 删除；拖拽排序
- **规则**：同步 C 端首页 Banner 轮播（**纯图，不叠加标签**）；**仅上线状态展示**；下线可删除，删除后序号重排

> 品牌信息（品牌名称、App 图标、Logo、品牌故事）由**前端配置文件**维护，不在后台配置。

### 8. 系统设置

- **配送设置**：基础运费、满额包邮门槛（同步 C 端运费计算与包邮判断）
- **支付设置**：微信支付开关、商户号、API 密钥（脱敏展示）
- **订单设置**：
  - 待付款超时时间：默认 30 分钟，可配置（10~120 分钟）
  - 自动确认收货天数：发货后 N 天自动确认收货，默认 15 天
- 保存成功 Toast，重置恢复上次保存值

### 9. 运维管理（仅超级管理员）

运维管理含 **4 个 Tab**：系统健康（技术）/ 运营监控（产品）/ 日志中心 / 数据操作。前两者把「偏开发的健康」与「偏产品的监控」拆开，互不干扰。

#### 9.1 系统健康（偏开发 / 运维）
- **服务状态卡**：服务 / 数据库 / Redis / 磁盘 状态（绿点正常 / 红点异常）
- **资源监控**：JVM 堆内存、CPU 使用率、系统负载，自动刷新
- **用途**：判断系统是否「活着、跑得动」，故障排查用

#### 9.2 运营监控（偏产品 / 业务）
- **流量指标**：PV / UV / 今日活跃用户 / 新增用户
- **业务实时指标**：实时下单数、实时 GMV、支付成功率、转化漏斗（浏览 → 加购 → 下单 → 支付）
- **趋势**：PV/UV 趋势折线（与数据看板的经营指标互补——看板看「生意结果」，这里看「流量与转化健康度」）
- **用途**：判断产品 / 运营是否健康，发现流量异常

#### 9.3 日志中心（两类日志分 Tab）
- **C 端业务日志**（监控 / 回溯）：记录 C 端用户的关键读写行为（下单 / 支付 / 退款申请 / 领券到账 / 登录等），含 用户 / 行为类型 / 关键参数 / 时间 / 结果。用于**业务监控与问题回溯**（如「某用户下单为何失败」）。支持搜索 / 类型筛选 / 日期范围 / 分页
- **管理员操作日志**（审计）：记录后台管理员的全部写操作（商品上下架 / 改价 / 发货 / 处理退款 / 改配置 / SQL 操作等），含 操作人 / 类型 / 内容 / 时间。用于**权限审计与追责**。支持搜索 / 类型筛选 / 日期范围 / 分页
- **区分理由**：C 端业务日志是「用户在前台干了什么」（量大、偏监控）；管理员操作日志是「管理员在后台改了什么」（偏审计），两者受众与用途不同，分开存储与查询

#### 9.4 数据操作
- SQL 查询（只读）直接执行
- SQL 修改需**二次确认**（显示 SQL + 影响行数预估）
- **禁止 DROP/TRUNCATE 等危险语句**；查询结果最多返回 500 行；所有数据操作记入**管理员操作日志**
