# 后台管理产品设计

## 一、概述

后台供管理员进行商城运营管理，覆盖商品、营销、订单、用户、内容、系统配置与运维等模块。

> C 端产品需求见《C端产品设计.md》；UI 设计见《后台管理UI设计.md》、《后台管理UI原型.html》。

## 二、角色与权限

后台**不区分角色**，统一为「管理员」单一角色，可访问全部模块：数据看板、商品、订单、用户、优惠券、内容、系统设置、运维管理（含系统状态 / 操作日志 / 数据操作）。

- 所有写操作记入操作日志（操作人 + 时间 + 类型 + 内容）
- 禁用用户后该用户无法登录商城
- **数据操作（SQL）改为只读查询**（独立只读 DB 账号 + 结果行数上限 + 全程记审计），不开放在线改 SQL；数据修复走 SSH + 事务或预置运维动作（见 §8.4）

## 三、模块清单

| 模块 | 主要职责 |
|------|---------|
| 数据看板 | 核心经营指标与趋势分析（含退款、优惠券核销数据） |
| 商品管理 | 商品列表（含编辑表单）、分类管理 / 标签管理（工具栏入口）、库存——**单级菜单** |
| 订单管理 | 订单查询、发货、物流单号、退款 / 售后处理 |
| 用户管理 | 用户列表、用户详情 |
| 优惠券管理 | 券模版的创建 / 编辑 / 停发、核销数据、**券发放明细**（用户券由系统下发） |
| 内容管理 | Banner |
| 系统设置 | 配送（运费 / 包邮门槛）、支付配置、订单超时配置 |
| 运维管理 | 系统健康、系统监控、操作日志、数据操作 |

> **说明**：本商城**不设会员体系、不设积分体系**——无会员等级、无升级送券、无下单折扣、无积分。优惠券面向全员 / 新人等标签人群统一下发。

## 四、各模块产品需求

### 1. 数据看板

> **定位（个体商户，简单直接实用）**：店主每天打开后台第一眼只需回答两件事——「**今天赚了多少**」和「**有什么活儿要干**」。因此看板砍掉企业级多维分析，只保留今日经营快照（含待办活儿）、一条趋势和热销榜。**不做**全局时间段切换、订单/退款/用户多张分析饼图、客单价 / 核销率等二级指标卡（这些不在看板重复堆叠）。

- **入口**：登录后默认首页 / 侧栏「数据看板」
- **今日概览（一行 3 张卡，今日固定快照，无全局时间切换）**：回答「赚了多少」——今日销售额、今日订单、今日新增用户，大数字 + 较昨日环比（↑绿 / ↓红）
- **待处理事项（一行）**：回答「有什么活儿要干」——待发货、退款售后，各显示数量并红色高亮 + 闪烁红点，点击分别跳订单管理「待发货」/「退款售后」Tab。只列店主真要动手的活（待付款等买家付钱、待收货等买家确认，店主无需操作，故不列）
- **销售趋势**：单张柱状图（日期 × 销售额），**每根柱顶标注当日金额**（如 ¥1,200），右上「近 7 天 / 近 30 天」切换，默认近 7 天；**这是页面唯一的时间维度入口**
- **热销商品 Top 5**：排行表（排名 / 商品名 / 销量 / 销售额），**商品名以蓝色链接样式呈现（提示可点击），点击该行跳商品列表并按该商品名预填搜索定位到该商品**；右上「查看全部」跳商品管理

### 2. 商品管理

> **导航说明**：商品管理为**单级菜单**（不再有「商品列表 / 分类管理」二级子菜单）。分类管理、标签管理入口收进商品列表页工具栏，作为「标签管理」「分类管理」按钮放在「+ 新增商品」旁边——分类与标签都是商品的属性维度，与商品同屏管理更顺手，也省去一层菜单跳转。

#### 2.1 商品列表
- **工具栏**：搜索（商品名）+ 分类筛选 + 状态筛选（上架 / 下架 / 售罄）；右侧操作区「标签管理」次按钮 +「分类管理」次按钮 +「+ 新增商品」主按钮

#### 2.1.1 商品编辑弹窗（新增 / 查看 / 编辑）

> **弹窗承载**：新增、查看、编辑统一走同一个**弹窗**（不跳独立页面），三态共用一套表单——新增为空表单、查看为只读态（字段灰显、底部仅「关闭」）、编辑为可写态。在商品列表点「+ 新增商品」「查看」「编辑」或在数据看板点热销商品行均打开此弹窗。

```
┌── 新增 / 查看 / 编辑商品 ───────────────────── ✕ ──┐
│ 商品名称  [____________________________]            │
│ 商品主图  [＋上传]  （1 张，建议 1:1）               │
│ 商品分类  [蜂蜜 ▾]                                   │
│ 商品标签  ○无 ○热销 ○新品 ○礼盒 …（单选，取自标签管理）│
│ 是否推荐  [○]  （开启进入 C 端首页精选推荐区）        │
│ 单次限购  [___] 件（0 或留空 = 不限购）               │
│ ── 规格与价格（最多 4 个）──                         │
│  规格名 [500g]  售价 [¥88]  库存 [150]   [删除]     │
│  规格名 [1kg ]  售价 [¥158] 库存 [80 ]   [删除]     │
│                                      [+ 添加规格]   │
│ ── 产品亮点（多条，首条复用为 C 端推荐种草语）──      │
│  · [纯天然零添加_________________]  [删除]          │
│  · [古法熬制工艺_________________]  [删除]          │
│                                      [+ 添加亮点]   │
│ 详情图    [＋上传多张]  （长图 / 多图，C 端详情页）   │
│                                   [取消] [保存]     │
└────────────────────────────────────────────────────┘
```

- **三态差异**：新增＝空表单；**查看＝全字段只读 + 底部仅「关闭」**；编辑＝预填当前值且可写。查看态下可一键切到编辑态
- **基础字段**：商品名称、主图（1 张）、分类（下拉）、标签（单选，可不选；选项取自「标签管理」启用的标签）、是否推荐（开关）、**单次限购**（整数，0 或留空＝不限购；C 端商品详情页加量达此上限时拦截并 Toast「该商品每单限购 N 件」）
- **规格与价格**：每条含 规格名 / 售价 / 库存；最多 4 个；单规格商品填一条即可（C 端规格区不展示）
- **产品亮点**：多条文本，**首条自动复用为 C 端精选推荐卡的种草语**（无需单独录入推荐语）；无亮点则 C 端不展示该行
- **详情图**：可上传多张，C 端商品详情页展示
- **无营销开关**：表单内不含任何营销 / 折扣开关，所有商品默认统一参与满减券与无门槛券
- **保存校验**：名称、主图、分类、至少 1 条规格（含售价、库存）为必填；保存成功 Toast 后关闭弹窗并刷新列表
- **列表字段**：主图、名称、售价、标签、是否推荐、库存、分类、状态、操作
- **库存预警**：库存 < 20 显示黄色「库存不足」；库存 = 0 显示红色「已售罄」并自动置为售罄状态
- **操作**：查看（只读详情）/ 新增 / 编辑 / 上架 / 下架；分页
- **商品字段**：名称、主图、售价（只售价无划线价）、规格（最多 4 个，可对应不同价格）、标签（取自「标签管理」，**单选，可不选**；用于 C 端商品卡片左上角角标，每个商品最多展示一个）、**是否推荐（用于 C 端首页精选推荐区）**、**单次限购（0＝不限）**、库存、分类、详情图、产品亮点
- **营销参与（统一规则，无单品开关）**：所有商品默认参与满减券与无门槛券，无需逐个配置开关；C 端的券标签、到手价计算对全部商品统一生效。**无任何商品营销 / 折扣开关**

#### 2.2 分类管理
- **入口**：商品列表页工具栏「分类管理」按钮 → 打开分类管理弹窗 / 抽屉（不占独立菜单）
- **字段**：分类名称、图标、排序、状态
- **操作**：新增 / 编辑 / 启用 / 禁用 / 删除；拖拽 `≡` 调整排序
- **规则**：
  - 同步到 C 端首页品类专区与商品列表筛选栏
  - **仅商品数量为 0 的分类可删除**
  - 禁用的分类 C 端不展示

#### 2.3 标签管理
- **入口**：商品列表页工具栏「标签管理」按钮 → 打开标签管理弹窗 / 抽屉（不占独立菜单）
- **字段**：标签名称、颜色（C 端角标底色）、排序、状态
- **操作**：新增 / 编辑 / 启用 / 禁用 / 删除；拖拽 `≡` 调整排序
- **规则**：
  - 标签列表即商品编辑表单「商品标签」单选项的来源（不再写死「热销 / 新品 / 礼盒 / 特惠」），新增标签后表单自动多一个可选项
  - 用于 C 端商品卡片左上角角标，**每个商品最多展示一个**
  - 禁用的标签不在商品表单中可选，已绑定该标签的商品 C 端不再展示该角标
  - **仅未被任何商品引用的标签可删除**

### 3. 订单管理

```
┌──────────────────────────────────────────────────────┐
│ 订单管理                                      [导出订单] │
├──────────────────────────────────────────────────────┤
│ [全部][待付款][待发货][待收货][已完成][已取消][退款售后]  │
│ [🔍订单号/买家姓名]                                     │
├──────────────────────────────────────────────────────┤
│ 订单号│买家│商品│金额│状态│下单时间│操作                │
│ ..01 │张三│百花蜜+2│¥78│待发货│06-10│ 详情  发货      │
│ ..98 │李四│蜂王浆 │¥168│退款售后│06-09│详情 处理退款   │
│  ▼ 展开行：收货人/地址/商品明细/优惠券/运费/实付/备注    │
└──────────────────────────────────────────────────────┘

  ┌── 发货弹窗（仅待发货）──┐   ┌── 处理退款弹窗（仅退款售后）──┐
  │ 物流公司 [顺丰 ▾]      │   │ 退款类型：仅退款 / 退货退款    │
  │ 物流单号 [SF________] │   │ 退款金额：¥168（红色大字）     │
  │        [取消][确认发货]│   │ 退款原因 / 补充说明 / 凭证图   │
  └──────────────────────┘   │      [拒绝(填理由)] [同意]    │
                             └──────────────────────────────┘
```

- **操作列按状态变化**：
  - 待付款 → 仅「详情」（不可代付）
  - 待发货 → 「详情」+「发货」
  - 待收货 / 已完成 → 「详情」（已完成可在详情内手动发起退款）
  - 退款售后 → 「详情」+「处理退款」（退货退款同意后变为「确认收货」）
- **状态 Tab**（附数量）：全部 / 待付款 / 待发货 / 待收货 / 已完成 / 已取消 / 退款售后
- **搜索**：订单号 / 买家姓名
- **列表**：订单号、买家、商品（首件 + 共 N 件）、金额、状态、下单时间、操作；点击行展开详情（收货人 / 地址 / 商品明细 / 优惠券 / 运费 / 实付 / **订单备注**）
- **导出订单**：导出当前筛选条件下的订单
- **发货**：仅待发货订单可操作，弹窗填写**物流公司 + 物流单号** → 转待收货；单号同步 C 端订单详情物流时间轴
- **退款 / 售后处理**（核心，防货财两空）：
  - **仅退款（未发货）**：待发货订单申请退款，商家审核通过后**直接原路退款**，无需寄回
  - **退货退款（已发货 / 已收货）**：流程「**提交申请 → 商家审核 → 买家寄回 → 商家确认收货 → 退款**」，**商家确认收货后才打款**
  - 「处理退款」支持：同意 / 拒绝、填写退款金额、记录买家寄回单号、确认收货并打款
  - 退款资金与权益：现金原路退回；优惠券若未过期原路退回
- **超时取消**：待付款订单超过 30 分钟未支付自动取消（超时时长可在系统设置中配置）
- **售后处理详细流程**：
  - 列表中退款售后订单展示**售后状态子标签**：待审核 / 已同意·待寄回 / 已寄回·待确认 / 已退款 / 已拒绝
  - **审核操作**：点击「处理退款」弹窗，展示买家申请信息（退款原因 / 退款金额 / 补充说明 / 凭证图片），操作「同意」/「拒绝」（拒绝需填写理由）
  - **仅退款（未发货）**：同意后系统自动原路退款，订单转「已退款」
  - **退货退款（已发货）**：同意后等待买家寄回 → 买家填写寄回单号后订单展示寄回物流信息 → 商家点击「确认收货并退款」→ 系统原路退款
  - **寄回单号以买家最新填写为准**：买家在「待商家收货」阶段可修改寄回单号，处理退款弹窗的寄回物流信息区**始终展示最新单号**（避免商家凭旧单号收货出错）
  - **退款资金与权益处理**：现金原路退回；未过期优惠券原路退回
  - **已完成订单售后（人工承接）**：C 端「申请售后」引导至微信客服会话，客服核实后由管理员在**该订单详情页「手动发起退款」**承接（填写退款金额与原因，原路退款）——即客服会话为入口、订单详情手动退款为处理点，无独立工单模块

### 4. 用户管理

> **无会员体系**：本商城不设会员等级 / 升级送券 / 积分，用户列表与详情均不展示等级、累计消费、订单数等会员相关字段。

- **搜索**：昵称 / 手机号
- **列表**：头像、昵称、手机号、注册时间、状态、操作
- **操作列**：「消费记录」「优惠券」「禁用」——头像 / 昵称 / 手机号 / 注册时间 / 状态列表已直观展示，故**不设「详情」聚合弹窗**，消费记录与优惠券各自直接打开独立弹窗（免去切 Tab）：
  - **消费记录弹窗**：该用户的订单列表（订单号 / 商品 / 金额 / 状态 / 时间），可点击查看订单详情；分页
  - **优惠券弹窗**：该用户持有的优惠券列表（券名 / 类型 / 来源：新人 / 活动 / 状态：可用·已使用·已过期 / 使用时间）；分页
  - 两弹窗均**不重复展示用户基本信息**（列表已展示），标题前缀该用户昵称
- **禁用**：切换状态，禁用后无法登录商城
- **新人定义**：注册 / 登录后**且无历史已支付订单**的用户（用于新人券发放资格校验）

### 5. 优惠券管理

> **核心模型：券模版（coupon）+ 用户券（user_coupon）。** 后台只创建并管理「券模版」（定义券的类型、面额、有效期、总量、**适用用户标签**等规则）；用户实际持有的是由系统**下发**生成的「用户券」实例。**全程系统下发、无手动领取（实质＝后端自动领）**——用户登录 / 进首页时前端调一次「查询可领券」接口，后端按券模版的**用户标签资格实时计算**并自动生成用户券，再把新到账券返给前端弹窗告知「已为你发放 N 张券」，按钮「去使用」（详见《C端产品设计.md》优惠券章节）。这样可避免「领券不下单」的羊毛账户占用名额与库存。

- **状态 Tab**：全部 / 进行中 / 未开始 / 已结束 / 已停发
- **筛选与分页**：状态 Tab 下提供券名搜索框做精筛；券模版较多时卡片网格底部分页（每页 12 张），状态筛选、券名搜索、分页三者叠加生效
- **券卡展示**（券模版维度）：券名、类型、有效期、限领、**已发 X/总量**、**已用 + 核销率**、状态、操作（编辑 / 停发 / 删除 / **发放明细**）
- **新建 / 编辑弹窗（券模版）**：「+ 新建券模版」与券卡「编辑」均打开同一弹窗（宽约 560px）。字段：券名、券类型（满减 / 无门槛）、优惠金额（满减券额外填门槛金额，无门槛券隐藏门槛输入）、有效期、每人限领张数、发行总量、**适用用户标签**（多选，不选＝通投全员）。**已开始（已有下发记录）的券模版编辑时，券类型 / 优惠金额 / 门槛等核心规则字段不可改**（避免已发出用户券的规则被篡改），仅可延长有效期、增加发行总量；保存校验券名 / 类型 / 优惠金额 / 有效期 / 限领 / 总量必填
- **适用用户标签（下发资格限制）**：多选标签（不选＝通投所有用户），由系统**实时函数计算**——如「新人」（无历史已支付订单）。用户登录 / 进首页查询可领券时，后端逐张判定其标签集是否命中模版要求的全部标签（AND 关系），命中才自动下发。**资格只在领取时刻校验**；券一经到账即为用户资产，后续标签变化（如新人下首单后不再是新人）**不影响已到账券的使用**。新增标签类型只需后台增加一种标签选项，无需改下发流程
- **券发放明细（券 → 用户的下发关系）**：券卡上「发放明细 ›」打开弹窗 / 抽屉，列出该券模版下发的全部用户券——列 **用户（昵称 / 手机号）/ 状态（未使用 / 已使用 / 已过期）/ 下发时间 / 使用时间**，支持按状态筛选 + 分页。与「用户管理 → 优惠券弹窗」（用户 → 券）互为反向视角，便于核查某张券发给了谁、用了多少（数据取自 user_coupon 按 coupon_id 查，无需新增表）
- **下发场景（系统按规则自动生成用户券，无手动领取）**：
  - **标签通投券（含新人券 / 活动券）**：用户登录 / 进首页查询时，后端按券模版「适用用户标签」实时计算资格自动下发——配「新人」标签即新人券，不配标签即全员活动券，未来可配其他标签做定向投放
- **核销数据**：已发数量 = 下发的用户券总数；已用数量 = 状态为「已使用」的用户券数；核销率 = 已用 / 已发
- **状态流转**：券模版四态——未开始 / 进行中 / 已结束 / 已停发。
  - **未开始 → 进行中 → 已结束**：由有效期（开始 / 结束时间）与当前时间**自动推进**，无需人工操作（now < 开始＝未开始；区间内＝进行中；超过结束＝已结束）
  - **停发**：唯一的人工流转，可在任意时刻点「停发」，**终态不可逆**；停发后不再下发新的用户券，**已发出的用户券不受影响**（各自用到过期）。停发优先级最高——即使仍在有效期内，停发后也立即显示「已停发」并停止下发
  - **仅「进行中」可下发**：未开始 / 已结束 / 已停发均不向用户下发新券
- **删除规则**：**仅「未开始且无任何下发记录」可删除**；一旦已开始（已有下发记录）只能停发，不能删除
- **券组合默认始终最优**：下单时系统自动从用户可用券中选「无门槛 + 满减各一张」抵扣最大的组合，无开关、无需配置

### 8. 运维管理

运维管理含 **4 个 Tab**：系统健康（技术）/ 系统监控（接口）/ 操作日志 / 数据操作。前两者把「系统活没活」与「接口忙不忙」拆开，互不干扰。

#### 8.1 系统健康（偏开发 / 运维）
- **服务状态卡**：服务 / 数据库 / Redis / 磁盘 状态（绿点正常 / 红点异常）
- **资源监控**：JVM 堆内存、CPU 使用率、系统负载，自动刷新
- **用途**：判断系统是否「活着、跑得动」，故障排查用

#### 8.2 系统监控（偏技术 / 接口）
> **指标源白嫖 Actuator / Micrometer + 既有 Redis 限流器（后端零额外埋点），但为看趋势曲线需落库。** 回答「接口忙不忙、慢不慢、有没有报错、有没有被刷」，与 8.1 系统健康（看系统活没活）互补；不做 PV/UV / 转化漏斗等产品流量分析（个体店流量小、样本无统计意义且埋点成本高，ROI 低，已砍）。

- **数据落库（支撑曲线）**：Micrometer 指标默认存于 JVM 进程内存（实时、不持久、重启清零、多实例各算各的），**无法直接出历史曲线**。故由定时任务（如每分钟）拉一次 Actuator `metrics` 关键值，快照插入 `metrics_snapshot` 表（时间戳 + 各指标值），后台按表聚合画**趋势折线**；当前实时值仍可直接拉 `metrics` 端点。限流拦截数据本就在 Redis（持久），可直接读或一并快照入表
- **接口 QPS & 耗时**：Top N 繁忙接口的请求量与 p95 耗时（源 `http.server.requests`），趋势曲线 + 当前值
- **错误率**：4xx / 5xx 占比与趋势 +「每分钟 ERROR 日志条数」（源 `logback.events{level=error}`），飙升即有异常
- **DB 连接池（HikariCP）**：active / idle / **pending**（等连接的线程数）——`pending > 0` 是连接耗尽/慢查询的早期预警，比堆内存更早暴露「接口集体卡死」
- **Tomcat 线程池**：busy / max——busy 逼近 max 说明处理线程被打满；与 QPS 配合可区分「流量大」还是「某接口慢拖垮线程」
- **GC 停顿**：`jvm.gc.pause` 次数与 max 停顿时长——抓「堆够大但频繁 Full GC 导致周期性卡顿」，光看堆占用看不出
- **异常流量（限流拦截）**：被限流拦截次数 / 触发限流的接口与来源（源 §9 既有 Redis 滑动窗口限流计数）——「有没有人在刷接口」的直接证据
- **用途**：单机生产环境故障预警与排障；指标采集零额外埋点，工作量仅 `metrics_snapshot` 表 + 定时快照任务 + 后台展示页

> **日志分层（关键边界）**：操作日志记录的是**结构化关键事件**（C 端下单 / 支付 / 退款 / 领券 / 登录、管理员写操作等节点），落 `operation_log` 表、量可控、可前台检索；**不入库的是全量请求明细**（每个请求的入参 / 第三方调用 / 异常堆栈，PV 量级）——这类走 Logback 文件日志（带 traceId），留在服务器 `logs/`，避免压垮业务库。前台靠 **traceId** 把两者串起来：在事件日志里拿某条的 traceId 去文件日志精确定位该次请求的逐行细节。

#### 8.3 操作日志（单一列表，按操作主体筛选）
> **不分 Tab、不拆表**：C 端业务日志与管理员操作日志同源同构（同存 `operation_log`，靠 `operator_type` 区分 1=C 端用户 / 2=管理员 / 3=系统），故合并为**一个列表 + 一个「操作主体」筛选**，避免同表查两次、来回切 Tab；排查问题时「用户下单失败 → 管理员随后手动退款」常是同一条因果链，合并后可跨主体看完整链路。

- **统一字段**：时间 / **操作主体**（C 端用户 / 管理员 / 系统）/ 操作人（昵称或 admin）/ 类型 / 内容 / 结果 / **traceId**
- **筛选与检索**：「操作主体」下拉（全部 / C 端用户 / 管理员 / 系统）+ 关键词搜索（操作人 / 内容）+ 类型筛选 + 日期范围 + 分页
- **按用户串轨迹（回溯）**：筛「C 端用户」+ 搜某 user_id，按时间序列出其全部关键事件，还原**业务动作链**（浏览→加购→下单→支付→…），定位「某用户下单为何失败」
- **审计追责**：筛「管理员」即得后台全部写操作（上下架 / 改价 / 发货 / 处理退款 / 改配置 / SQL）记录
- **traceId 导航**：每条带 traceId，需看该次请求的全量技术明细（参数 / 堆栈）时，以 traceId 去文件日志定位（前台不堆全量原始日志）

#### 8.4 数据操作（只读查询）
> **只读定位**：本模块仅做**线上数据查询排障**（如核对某用户的券 / 订单状态、定位数据异常），**不提供在线 SQL 修改**。改数据走更安全的路径，见下「数据修复」。

- **只读查询**：输入 SELECT 直接执行，结果表格展示
- **真·只读（账号级隔离，非关键字黑名单）**：查询走一个**独立的只读 DB 账号**（仅授予 SELECT 权限），从数据库权限层面杜绝任何写操作——而非靠应用层关键字黑名单拦截（黑名单挡不住注释绕过、大小写变体、缺 WHERE 的写语句等）
- **结果上限**：单次查询最多返回 **500 行**（防大表全捞拖垮库 / 撑爆前台）
- **审计**：每次查询记入**操作日志**（操作人 + 时间 + SQL + 行数）
- **数据修复（不在后台做）**：确需改线上数据时，走 **SSH 登机 + DB 客户端开事务手动改**（可 BEGIN→核对影响行→COMMIT/ROLLBACK），或为高频运维场景在后台预置**具体的运维动作按钮**（语义明确、参数受控、带二次确认与审计），避免开放式在线改 SQL 的高风险面（缺 WHERE 误伤全表、绕过业务校验产生脏数据、账号被盗即可任意改库）
