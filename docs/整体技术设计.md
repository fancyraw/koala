# 整体技术设计

> 本文档基于《C端产品设计.md》《后台管理产品设计.md》对齐最新需求，给出**可在正式环境部署、通用可扩展**的工程方案。覆盖：分层架构与扩展点、数据模型、核心算价/下单/支付/售后/券机制、幂等并发、安全、可观测性与部署运维。
>
> 业务基线：**无会员体系、无积分体系**——无会员等级、无升级送券、无下单折扣、无积分。**全部商品统一参与满减券与无门槛券，无单品营销开关**；优惠券由系统统一向全员自动下发，无人群定向 / 标签机制，每模版每人 1 张。

---

## 一、设计目标与原则

### 1.1 目标
- **生产可部署**：多环境隔离、密钥外置、HTTPS、健康检查、可观测、可备份恢复、可灰度回滚。
- **通用可扩展**：核心能力以接口抽象（支付/存储/物流/登录/通知），新增渠道不改业务代码；业务策略走配置中心，改参数不发版。
- **数据一致**：资金与库存/券资产强一致，幂等防重，事务边界清晰。

### 1.2 架构原则
| 原则 | 落地方式 |
|------|---------|
| 分层清晰 | Controller → Service → Mapper(持久层) 三层，依赖单向向下 |
| 面向扩展 | 易变点定义 SPI 接口 + 多实现，由配置选择激活实现（策略模式 / 工厂） |
| 配置外置 | 环境差异走 Spring Profile + 环境变量；业务策略走 `sys_config`（运行期可调） |
| 事件解耦 | 订单支付成功等关键节点发应用事件，副作用（销量统计、日志、预留通知）由监听器处理 |
| 约定优于配置 | 统一响应体、统一错误码、统一分页、统一鉴权与日志切面 |
| 安全默认 | 默认最小权限、默认脱敏、默认参数校验、默认幂等 |

## 二、整体架构

### 2.1 系统组成

```
                          用户微信小程序        运营/客服浏览器
                                │                    │
                          ┌─────▼────────────────────▼─────┐
                          │        Nginx 反向代理            │  :443 (TLS)
                          │  共用域名,本应用统一挂 /koala 命名空间 │
                          │  /koala/          → 后台管理 Web 静态 │
                          │  /koala/api/v1/    → 后端服务(strip前缀)│
                          │  /koala/api/v1/admin→ 后端服务(后台)  │
                          └─────────────┬───────────────────┘
                                        │
                              ┌─────────▼─────────┐
                              │   后端服务 Spring   │  :8080 (可水平扩展 N 实例)
                              │  Controller/Service │
                              │  Mapper(持久层)      │
                              │  + Actuator 健康检查 │
                              └──┬──────┬──────┬────┘
                  ┌──────────────┘      │      └──────────────┐
            ┌─────▼─────┐         ┌──────▼──────┐       ┌──────▼───────┐
            │   MySQL    │         │    Redis     │       │  外部服务抽象  │
            │ 主(+可选从) │         │ 缓存/锁/幂等  │       │ 支付/存储/物流 │
            └───────────┘         └─────────────┘       └──────────────┘
```

- **无状态后端**：会话用 JWT，服务器不存会话态，后端可多实例水平扩展（Nginx 轮询 / 最少连接）。
- **共享态下沉 Redis**：分布式锁、幂等令牌、限流计数、热点缓存全部放 Redis，保证多实例一致。

### 2.2 技术栈

| 层级 | 选型 | 说明 |
|------|------|------|
| C 端 | uni-app（Vue3） | 编译为微信小程序 |
| 后台管理 | Vue3 + Element Plus + Vite | SPA，Nginx 托管静态产物 |
| 后端 | Spring Boot 2.7 + JDK 8 | RESTful；MyBatis-Plus 持久层 |
| 数据库 | MySQL 5.7（InnoDB / utf8mb4） | 业务主存储；Flyway 管理 schema 迁移 |
| 缓存 | Redis 7 | 会话黑名单、幂等、分布式锁(Redisson)、限流、热数据 |
| 连接池 | HikariCP | 默认连接池 |
| 文件存储 | 对象存储抽象（七牛 / 阿里 OSS / S3 任选） | 默认七牛，接口隔离可切换 |
| 支付 | 支付渠道抽象（默认微信 JSAPI） | `PaymentChannel` 接口，可扩展支付宝等 |
| 物流 | 物流查询抽象（默认快递100） | `LogisticsProvider` 接口，可扩展快递鸟 |
| 登录 | 登录方式抽象（C 端微信、后台微信扫码） | `user_auth` 多端凭证表支撑 |
| 可观测 | Spring Actuator + 文件日志（带 traceId）+ operation_log 表 | 个人开发者轻量方案，不引 Prometheus/Grafana/ELK |
| 部署 | Docker + Docker Compose（阿里云 2C2G 单机） | 本地构建打镜像 → `docker save`+ssh 传服务器运行，无需 CI/CD、无需镜像仓库 |

│   │   ├── service/            # 业务逻辑层(接口 + impl)

> **分层原则**：传统三层（Controller → Service → Mapper），按技术层组织包，依赖单向向下；同类代码集中（所有 controller 一处、所有 service 一处），简单直观。Service 间可直接相互调用。

## 三、扩展性设计（核心：易变点抽象）

把「会随渠道/供应商变化」的能力定义为接口，业务代码只依赖接口；具体实现由配置选择，新增渠道**不改调用方**。

### 3.1 支付渠道抽象

```java
public interface PaymentChannel {
    String code();                                   // wechat / alipay ...
    PrepayResult prepay(PrepayCommand cmd);          // 统一下单，返回前端调起参数
    NotifyResult parseNotify(HttpServletRequest r);  // 解析+验签回调
    RefundResult refund(RefundCommand cmd);          // 退款
}
// WechatPayChannel 实现微信 JSAPI；扩展支付宝只需新增 AlipayChannel 并注册
// 通过 PaymentChannelFactory.get(channelCode) 路由，channelCode 存于 payment 表
```

### 3.2 文件存储抽象

```java
public interface FileStorage {
    UploadCredential issueCredential(String bizType);  // 前端直传凭证
    String publicUrl(String key);                      // 拼访问 URL
    void delete(String key);
}
// 默认 QiniuStorage；切换 OSS/S3 改配置 koala.storage.provider 即可
```

### 3.3 物流查询抽象

```java
public interface LogisticsProvider {
    List<TrackNode> query(String company, String trackingNo);  // 轨迹节点(最新在前)
}
// 默认 Express100Provider；结果带缓存层，详见 4.8
```

### 3.4 告警通知抽象

```java
public interface AlertNotifier {
    void send(AlertLevel level, String title, String content);  // 推送一条告警
}
// 默认 WeworkBotNotifier：POST 企业微信群机器人 webhook(markdown 卡片)
// webhook URL 走环境变量 koala.alert.wework-webhook(密钥不入库)；换钉钉/飞书只需新增实现
// 调用方：①ApplicationEvent 监听器(支付失败/退款异常/关键错误) ②阈值巡检定时任务(见 7.4)
// 防刷屏：发送前按告警类型在 Redis 置 5min 去重 key
```

### 3.5 业务策略走 sys_config
运费 / 包邮门槛、支付商户号、订单超时分钟数等**易调策略**统一进 `sys_config` 表，应用启动加载到本地缓存，后台/接口变更后失效刷新（见 7.3）。**改策略不发版**。

### 3.6 应用事件（解耦副作用，便于增量扩展）

```
OrderPaidEvent  → [销量+1] [业务日志] [预留:推送通知]
RefundedEvent   → [券原路退回] [业务日志]
```
基于 Spring `ApplicationEvent`；新增副作用只加监听器，不动主流程。

## 五、数据库设计

> 规范：引擎 InnoDB、字符集 utf8mb4；金额 `DECIMAL(10,2)`；时间统一 `created_at/updated_at`；并发更新的表加乐观锁 `version`；后台维护的表加 `created_by/updated_by` 审计字段。Schema 由 **Flyway** 版本化管理，正式环境**只增量、禁手改**。

### 5.1 用户与登录

```sql
-- 用户表（无会员体系、无折扣、无积分）
CREATE TABLE user (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键',
    nickname        VARCHAR(64)  DEFAULT '' COMMENT '昵称',
    avatar_url      VARCHAR(512) DEFAULT '' COMMENT '头像URL',
    is_valid        TINYINT      DEFAULT 1 COMMENT '是否有效:1=是 0=否',
    created_at      DATETIME     DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at      DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) COMMENT='用户表';
-- 无会员等级 / 累计消费 / 折扣 / 积分。优惠券通投全员、无新人/标签机制

-- 用户登录凭证表（多端可扩展：当前 wechat_mp，未来 email/apple/…）
CREATE TABLE user_auth (
    id         BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键',
    user_id    BIGINT NOT NULL COMMENT '用户ID',
    auth_type  VARCHAR(16)  NOT NULL COMMENT '凭证类型:wechat_mp/email',
    auth_id    VARCHAR(128) NOT NULL COMMENT '凭证标识:openid/邮箱',
    created_at DATETIME     DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    UNIQUE KEY uk_type_id (auth_type, auth_id),
    INDEX idx_user_id (user_id)
) COMMENT='用户登录凭证表';

-- 用户地址表
CREATE TABLE user_address (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    name VARCHAR(32) NOT NULL COMMENT '收货人姓名', 
    phone VARCHAR(20) NOT NULL COMMENT '收货人手机号',
    province VARCHAR(32) NOT NULL COMMENT '省', 
    city VARCHAR(32) NOT NULL COMMENT '市', 
    district VARCHAR(32) NOT NULL COMMENT '区/县',
    detail VARCHAR(256) NOT NULL COMMENT '详细地址', 
    full_address VARCHAR(512) NOT NULL COMMENT '完整地址(冗余,省市区+详细)',
    is_default TINYINT DEFAULT 0 COMMENT '是否默认地址:1=是 0=否',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    INDEX idx_user_id (user_id)
) COMMENT='用户地址表';
```

    product_id BIGINT NOT NULL COMMENT '商品ID',

    stock INT DEFAULT 0 COMMENT '库存',

### 5.3 优惠券（券模板 + 用户券）

```sql
-- 券模板（规则定义）：后台只管模板，用户券由系统下发生成
CREATE TABLE coupon (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键',
    name VARCHAR(64) NOT NULL COMMENT '券名称',
    type TINYINT NOT NULL COMMENT '1=满减券 2=无门槛券',
    discount_amount DECIMAL(10,2) NOT NULL COMMENT '优惠金额',
    min_spend DECIMAL(10,2) DEFAULT 0.00 COMMENT '满减门槛(无门槛=0)',
    total_count INT NOT NULL COMMENT '发行总量',
    issued_count INT DEFAULT 0 COMMENT '已发(下发的用户券数)',
    used_count INT DEFAULT 0 COMMENT '已用',
    valid_days INT NOT NULL COMMENT '领取后有效天数(到期=领取时间+N天)',
    is_valid TINYINT DEFAULT 1 COMMENT '是否有效:1=正常 0=已停发',
    version INT DEFAULT 0 COMMENT '乐观锁(防超发)',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) COMMENT='券模板表(已开始即有下发记录的不可删除,只能停发)';

-- 用户券（实例，含锁定态支撑资产冻结释放）
CREATE TABLE user_coupon (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    coupon_id BIGINT NOT NULL COMMENT '券模板ID',
    status TINYINT DEFAULT 0 COMMENT '0=未使用 1=已使用 2=已过期 3=已锁定(下单未支付)',
    lock_order_no VARCHAR(32) DEFAULT NULL COMMENT '锁定/使用的订单号',
    expire_at DATETIME NOT NULL COMMENT '到期时间(冗余,临期倒计时与过期任务)',
    granted_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '下发时间',
    used_at DATETIME DEFAULT NULL COMMENT '使用时间',
    UNIQUE KEY uk_user_coupon (user_id, coupon_id) COMMENT '每模版每人至多1张,兜底防并发重复下发',
    INDEX idx_user_status (user_id, status),
    INDEX idx_expire (status, expire_at)
) COMMENT='用户优惠券表';
```

## 六、核心模块设计

### 6.1 登录与鉴权

```
C端：wx.login() 取 code → /api/v1/user/login
   → code 换 openid → 查 user_auth(wechat_mp, openid)
     命中=登录；未命中=单事务创建 user + user_auth(自动注册)
   → 签发 JWT(含 userId、端类型)；触发自动领券查询(通投全员、每模版每人1张,见 6.6)
后台：仅微信扫码 → code 换 openid → 匹配 admin(wx_openid) 且 is_valid=1 → 签发 JWT(含 adminId)
   → openid 无匹配/未启用=拒绝登录(后台无注册口,管理员只能由超管邀请,见 6.10)
鉴权：网关切面校验 JWT；后台仅「超管 / 普通管理员」两档——业务模块两者皆可，**运维管理(系统健康/趋势/操作日志/只读SQL)仅超管**(JWT 内 isSuper 标记，切面校验 /admin/ops/**)；菜单按角色过滤、接口侧二次校验
JWT：短期 access + Redis 维护登出黑名单；禁用用户即时失效(is_valid=0 校验)
```

### 6.2 价格计算（与产品口径强一致）

**固定优先级逐级扣减，金额均不为负**。所有商品统一参与满减/无门槛券，**无任何折扣**：

```
1. 商品合计 = Σ(sku.price × qty)
2. 券抵扣(无门槛 + 满减 各一张叠加)
   - 满减券：门槛以「商品合计」判断并按面额直减
   - 无门槛券：直接按面额直减
   - 券后小计 = 商品合计 − (无门槛面额 + 满减面额)，下限 0
3. 运费：满额包邮门槛以「商品合计」判断；运费不参与券抵扣
4. 实付 = 商品合计 − 券抵扣 + 运费（微信支付要求最低 0.01）
```

- `/api/v1/order/preview` 与 `/submit` **共用同一算价服务**，保证预览=下单。
- **券组合默认始终最优**：系统在用户可用券中自动选「无门槛 + 满减各一张」抵扣最大的组合，无开关。
- 商详页只返回售价，不算「到手价」；券优惠统一在购物车/确认页体现。

### 6.3 下单 + 资产冻结（先锁定再支付，防货财两空）

```
提交订单（单事务 + 用户维度分布式锁防并发）：
  a. 幂等：前端 submitToken 经 Redis SETNX，重复提交拦截
  b. 扣库存：乐观锁 UPDATE stock=stock-N WHERE id=? AND stock>=N（影响0行=库存不足）
  c. 算价(6.2)，落 order 各金额字段
  d. 锁定券：user_coupon 0→3(锁定)、lock_order_no=订单号；写 order_coupon
  e. 建 order(status=0) + order_item
  f. expire_at = now + 超时分钟；返回订单号
  → 调支付渠道 prepay()，前端 wx.requestPayment()
```

### 6.4 支付结果处理（释放/核销，最高优先级）

```
支付成功回调 /api/v1/order/pay-notify（验签 → 幂等：transaction_id 唯一 + 状态 CAS）：
  - order 0→1(待发货)（CAS，已处理直接返回成功）
  - user_coupon 3→1(已使用)、coupon.used_count+1
  - payment 置成功
  - 发 OrderPaidEvent → 监听器：销量+1、业务日志

支付失败/取消/超时(定时任务)→ 释放资产（单事务）：
  - 回滚库存 stock+=N；释放券 user_coupon 3→0；order→4(已取消)
⚠ 资产释放必须与失败/取消同事务、最高优先级，杜绝「券没了」投诉
```

### 6.5 售后 / 退款

```
状态机：0待审核 →1通过待寄回 →(退货退款)2买家已寄回 →3商家已收货 →4已退款
                                └(仅退款,未发货)──────────────────→4已退款
                     └5已拒绝
仅退款(未发货)：商家审核通过 → 直接原路退款 → 4
退货退款(已发货)：审核通过 → 买家寄回(填单号) → 商家确认收货 → 退款 → 4
退款处理(事务)：调渠道 refund()→payment=3；未过期券原路退回 user_coupon→0；
              发 RefundedEvent → 按策略回滚销量
```

### 6.5.1 券模版状态流转（无时间窗，靠人工停发 + 发行量控制）

> 券模板**没有投放时间区间**——创建即生效，有效期是「领取后 N 天」（`valid_days`），由每张用户券各自计算到期，模板本身不设 start/end。是否还在下发由两件事决定：`status`（人工停发开关）与 `issued_count < total_count`（发行量是否发完）。

```
模板生命周期（无定时任务、无时间派生）：
  status=1 且 issued_count<total_count → 进行中（绿）  ◀── 可下发
  status=1 且 issued_count>=total_count → 已发完（灰）  ◀── 发行量耗尽,自动停止下发
  status=0                            → 已停发（灰,终态·不可逆）

人工流转（唯一写操作）：
  任意时刻点「停发」→ status=0 已停发
```

- **进行中**：创建后即进行中，无「未开始」概念；发行量发完自然停止，无需人工干预
- **停发**：唯一人工流转，置 `status=0`，**终态不可逆**；停发后不再生成新 user_coupon，**已发出的用户券不受影响**（各自按领取后 N 天用到过期）
- **可下发条件**：[6.6](#) 候选券判定即 `status=1 且 issued_count<total_count`
- **删除约束**：仅「无任何下发记录（issued_count=0）」可删；一旦有下发记录只能停发（见 `coupon` 表注释）

### 6.6 统一领券弹窗（系统下发，无手动领取 = 后端自动领）

> **「无手动领取」的技术实质 = 后端自动领。** 前端不调「领取」动作，登录/进首页时调一次「查询可领券」接口，后端把所有可下发（status=1 且未发完）且该用户尚未领过的券模板各发 1 张并**自动生成 user_coupon**，再把本次新到账的券返回给前端弹窗告知。用户只感知「已为你发放 N 张」。**通投全员、无标签 / 资格判定、每模版每人 1 张。**

```
登录/进首页 → /api/v1/coupons/auto-grant（查询并自动领，幂等）：
  1. 候选券模板：status=1正常、issued_count<total_count
  2. 去重过滤：该用户对此券未领过（user_coupon 不存在 user_id+coupon_id 记录；
     领过即不再发，含已用/已过期，无需算资格、无标签）
  3. 对通过的券：用户维度分布式锁 + 单事务下发
     - 用户级幂等键(user:{id}:coupon:{couponId}) 防并发重复领
       (配合 user_coupon 唯一索引 uk(user_id,coupon_id) 兜底)
     - coupon.version 乐观锁 + issued_count<total_count 条件更新(防超发)
     - 建 user_coupon(status=0)、coupon.issued_count+1
     - expire_at = now + valid_days（领取后 N 天到期）
  4. 返回本次新到账券列表 → 前端弹「领券弹窗」，按钮「去使用」
异常兜底：
  - 接口失败本次不弹、下次再查(前端不阻塞浏览)
  - 部分下发失败：已成功的正常返回，失败的下次重试
```

> **券一经下发即为用户资产。** user_coupon 下发后归用户所有，券模板停发 / 结束**不影响已到账券的使用**，与「已发出的用户券不受影响、杜绝券没了投诉」一致。

### 6.9 维护模式（一键停服公告）
后台一键开关，开启后锁定 C 端所有写操作并提示「系统维护中」，无需发版重启。

```
配置：sys_config system.maintenance_mode(0/1) + maintenance_notice(文案)
  → 后台 /admin/config/save(group=system) 写入即刷新本地缓存(7.3)，秒级生效
拦截：鉴权切面后置「维护模式拦截器」，读本地缓存判断
  - 关(0)：放行
  - 开(1)：拦截 C 端「写操作」(POST，下单/支付/购物车/售后/地址等)
           → 返回 5xxx 维护错误码 + maintenance_notice 文案，前端弹窗提示
  - 放行项：C 端 GET 查询(可浏览不可下单)、全部 /admin/* (后台需照常运维/关开关)
           → 支付回调 /pay-notify 例外放行(三方已扣款,不可拒,保证资产一致)
```

- 落在网关切面，集中一处控制；判断走本地缓存(7.3)，无 DB/Redis 额外开销。
- 开关本身就是 `sys_config` 的典型用例：**后台改、内存即时生效、不发版**。

│            开启后 C 端将无法下单/支付等操作      │
│  提示文案   [ 系统维护中,请稍后再试        ]    │  ← Input
│                              [ 保存 ]         │
└────────────────────────────────────────────┘
开关切换 / 文案保存 → POST /admin/config/save(group=system) → 秒级生效
开启时卡片顶部红色 Alert 常驻提示「当前处于维护中,C 端已停止下单」
```
- 开关 `el-switch` 绑定 `maintenance_mode`，二次确认弹窗防误触（开/关都弹 `el-messagebox`）。
- 文案 `el-input` 绑定 `maintenance_notice`，与开关同一表单一并提交。
- 维护中时后台仍可正常操作（拦截器放行 `/admin/*`），运营据此再关闭开关恢复。

### 6.10 管理员初始化与邀请（纯扫码，无密码/无注册口）

**超管初始化（鸡生蛋）**：超管微信 openid 写配置（`koala.admin.super-openid`，值走 `.env` 注入），启动时 `ApplicationRunner` 幂等建号——表中无此 openid 则建一行（`is_super=1`、`is_valid=1`，昵称/头像首次扫码登录时取微信资料回填）。配置一填、启动即有超管，**无需任何注册入口**。

**邀请新管理员（超管生成链接 → 新人扫码 → 待审核 → 超管启用）**：
```
1. 超管「管理员管理」点「+ 邀请」
   → 后端生成一次性 invite_token 存 Redis(TTL 30min)，返回邀请二维码/链接
2. 新人微信打开链接 → 扫码授权 code 换 openid
   → /admin/admins/accept 校验 token(有效且未用过) + openid 未注册
   → 建 admin 行(is_valid=0 待审核)、token 标记已用(一次性)
3. 超管在列表见到待审核项 → 点「启用」置 is_valid=1
   → 该新人此后即可扫码登录
```
- **一次性 + 短 TTL + 超管手动触发**：杜绝固定链接=公开注册口的风险；token 用过即废、转发也只能用一次。
- **扫完不直接生效**：新号落地即 `is_valid=0`，必须超管启用，后台权限不会因 token 误转发而旁落。
- **openid 一致性**：邀请扫码与登录扫码必须同一微信应用（开放平台同一 app），否则同一人两处 openid 对不上。
- **兜底**：超管微信丢失 → SSH 改 `.env` 的 `super-openid` 重启重新认领（符合「数据修复走 SSH」约定）。

## 七、关键技术机制

### 7.1 幂等与防重

| 场景 | 方案 |
|------|------|
| 提交订单 | 前端 submitToken(一次性) + Redis SETNX；按钮点击即置灰 loading |
| 支付回调 | transaction_id 唯一键 + 订单状态 CAS，重复回调返回成功 |
| 确认收货 | 状态机 CAS(status=2→3，影响0行视为已处理) + 前端置灰 |
| 自动领券 | 用户级幂等键 + user_coupon 唯一索引(每模版每人1张) + 发行量乐观锁；通投全员、无资格判定 |
| 售后申请 | order_no 维度当前状态校验，重复申请拦截 |
| 通用写 | 优先数据库唯一约束/乐观锁，不依赖前端 |

### 7.2 并发与一致性
- **库存防超卖**：乐观锁 `UPDATE … WHERE stock>=N`，无行锁竞争、无超卖。
- **券防超发**：`coupon.version` 乐观锁 + `issued_count<total_count` 条件更新。
- **分布式锁**：Redisson 按 `user:{id}` 锁下单关键段，多实例下仍互斥。
- **事务边界**：资金/库存/券变更同事务；第三方调用（支付/退款）放事务外或用「本地消息+对账」避免长事务。

### 7.3 缓存策略
| 数据 | 策略 |
|------|------|
| 首页聚合/分类/Banner | Redis 缓存 + 后台变更主动失效 |
| sys_config 业务策略 | 启动加载本地缓存，变更发失效信号刷新 |
| 物流轨迹 | 短 TTL（30min） |
| 防击穿 | 热点 key 互斥重建；空值短缓存防穿透 |

### 7.4 定时任务
| 任务 | 频率 | 说明 |
|------|------|------|
| 订单超时取消 | 每 1 分钟 | status=0 且 expire_at<now → 释放库存/券 |
| 优惠券过期 | 每天凌晨 | user_coupon 到期 → status=2 |
| 监控指标快照 | 每 1 分钟 | 拉 Actuator metrics → 写 metrics_snapshot（6 列：QPS/p95/错误率/连接池等待/CPU/堆，支撑趋势曲线） |
| 阈值告警巡检 | 每 1 分钟 | 读 metrics_snapshot 最新行 + Redis 限流计数，超阈值经 `AlertNotifier` 推送（Redis key 5min 去重） |
| 监控数据清理 | 每天凌晨 | 删除 metrics_snapshot 超过保留期（如 N 天）的旧快照 |
| 数据库备份 | 每天 3:00 | mysqldump→gzip→对象存储，留 30 天 |

> 多实例下定时任务需防重复执行：用分布式锁或 ShedLock 保证集群内单次执行。

## 九、接口规范

### 9.1 通用约定
- **路径前缀**：`/api/v1/`，版本化便于不破坏式升级；后台 `/api/v1/admin/`。
- **统一响应体**：
  ```json
  { "code": 0, "message": "ok", "data": {}, "traceId": "..." }
  ```
  `code=0` 成功；非 0 为业务错误码（分段：1xxx 鉴权、2xxx 参数、3xxx 业务、5xxx 系统）。
- **分页**：`?page=1&size=20`，返回 `{ list, total, page, size }`。
- **错误处理**：全局异常处理器统一兜底，区分业务异常（可展示文案）与系统异常（脱敏 + 告警）。
- **校验**：入参 Bean Validation；金额/状态等服务端二次校验，不信前端。

> **方法约定**：全站只用 **GET（查询）** 与 **POST（写操作）**，不用 PUT/DELETE。更新走 `/update`、删除走 `/delete` 等动词化路径，个人项目更省心、网关/代理兼容性也更好。
>
> **路径收敛（无路径变量）**：所有资源 ID / 业务号一律走 **query 参数**（GET）或 **请求体**（POST），**禁止 `/{id}`、`/{no}` 等路径变量**。例如详情用 `/products?id=123`、订单操作用 `/orders/cancel`（`no` 放 body）。好处：URL 模式有限且固定，便于 Nginx / 网关做**白名单精确匹配**与限流，无需正则段匹配，路由表收敛、鉴权切面也更易枚举。

## 十、可观测性（个人开发者轻量方案）

> 不引 Prometheus / Grafana / ELK 等独立基建——单机个人项目用 **Actuator + 文件日志 + 数据库表**就够，运维数据直接由后台「运维管理」页查接口/查表展示。

| 维度 | 方案 | 对应后台 |
|------|------|---------|
| 健康检查 | Spring Actuator `/actuator/health`（DB/Redis/diskSpace 探针）+ `/actuator/metrics`（JVM/CPU/内存），Nginx/容器探活 | 运维-系统健康(实时状态卡片) |
| 系统监控 | Actuator `http.server.requests`(QPS/p95/错误率) + HikariCP 等待数 + CPU/堆内存；定时任务每分钟快照入 `metrics_snapshot`(6 列精简)以出趋势曲线 | 运维-系统健康(健康卡片下方 6 项趋势图,与健康同页) |
| 日志 | Logback 文件日志 + MDC traceId 全链路串联，按天滚动 + 大小切分保留 N 天 | 服务器 `logs/` 目录 |
| 业务/审计日志 | operation_log：operator_type=1 C端业务、=2 管理员审计、=3 系统；带 traceId 关联文件日志 | 运维-操作日志(单列表,按主体筛选) |
| 告警 | 轻量：统一 `AlertNotifier` SPI，默认实现走**企业微信群机器人 webhook**(URL 走环境变量)。两类触发：①事件驱动——支付失败/退款异常/关键错误在 ApplicationEvent 监听器里直接推；②阈值巡检——定时任务读 `metrics_snapshot` 最新行,超阈值(错误率/p95/CPU/限流次数)推送。同类告警用 Redis key 5 分钟去重防刷屏。无独立告警系统 | — |

> **系统健康零额外成本**：服务 / DB / Redis / 磁盘状态与 JVM / CPU / 内存指标全部取自 Actuator `health` / `metrics` 端点，引入 `spring-boot-starter-actuator` 后开箱即有，**后端无任何额外埋点**，后台「系统健康」页仅做接口拉取 + 展示。Docker 单机部署下 Actuator 看的是容器内视角（堆内存、容器挂载磁盘），足够回答「服务活着、堆没爆、盘没满」；**不监控宿主机级指标**，避免引额外 agent。`/actuator/**` 端点须走管理员鉴权或限内网/容器网络访问，不裸暴露公网。

## 十一、部署与运维

### 11.1 多环境
`dev / test / prod` 三套 Spring Profile；差异（DB、Redis、密钥、域名、超管 openid `KOALA_SUPER_OPENID`、告警 webhook `KOALA_ALERT_WEWORK_WEBHOOK`）全走**环境变量 / Docker Secret**，镜像一份、配置分离，支持同镜像多环境部署。启动时按 `super-openid` 幂等建超管（见 6.10）。

### 11.2 Docker Compose（单机起步，2C2G 经济型 ECS）

> **镜像分发模型（非服务器本地编译）**：后端镜像在**本地构建**后，用 `docker save | ssh … docker load` 直接传到服务器加载、**不经任何镜像仓库**，服务器只负责运行、**不编译**。原因：`mvn package` + `docker build` 峰值吃 1.5G+ 内存，在已跑 MySQL/Redis 的 2C2G 机器上会 OOM；把编译挪到本地后，2G 跑起来绰绰有余，也省掉 ACR 开通与登录维护。

```yaml
services:
  mysql:
    image: mysql:5.7
    env_file: .env
    command: ["--innodb-buffer-pool-size=256M"]   # 2G 机器收紧 MySQL 内存,留给应用
    volumes: ["./data/mysql:/var/lib/mysql"]
  redis:  { image: redis:7-alpine, command: ["redis-server","--appendonly","yes"], volumes:["./data/redis:/data"] }
  server:
    image: koala/server:${SERVER_TAG}  # 本地 docker load 进来的镜像,服务器不 build
    env_file: .env                 # DB/Redis/微信密钥/JWT密钥 等敏感项
    depends_on: { mysql: {condition: service_healthy}, redis: {condition: service_started} }
    healthcheck: { test: ["CMD","curl","-f","http://localhost:8080/actuator/health"], interval: 30s }
    deploy: { replicas: 1 }        # 2G 单机跑 1 副本;内存充裕后再加(无状态,可水平扩展)
  nginx:
    image: nginx:alpine
    ports: ["80:80","443:443"]
    volumes: ["./deploy/nginx.conf:/etc/nginx/nginx.conf:ro","./deploy/certs:/etc/nginx/certs:ro"]
    depends_on: [server]
```

> 数据库迁移由 **Flyway** 在 server 启动时自动执行，保证 schema 与代码版本一致。
> `SERVER_TAG`（镜像版本，取 git commit 短 hash）放 `.env`，由部署脚本写入；2G 机器建议另配 **2G swap** 兜底突发峰值。

### 11.3 Nginx 路由

> **共用域名 + `/koala` 命名空间**：域名为多应用共用，本应用所有外部路径统一挂 `/koala` 前缀。前缀只在 Nginx 这一层做（方案 A）——转发后端时 strip 掉 `/koala`，**后端代码仍认 `/api/v1/`、零改动**，未来换前缀也只动 Nginx。后台静态产物同样挂 `/koala/`。

```nginx
# 后端 API：剥掉 /koala 前缀后转发，后端无感知
location /koala/api/v1/ {
    limit_req zone=api burst=20 nodelay;          # /api 限流兜底
    rewrite ^/koala(/.*)$ $1 break;               # /koala/api/v1/x → /api/v1/x
    proxy_pass http://server_upstream;            # :8080 多实例 upstream 负载
    proxy_set_header X-Forwarded-Prefix /koala;   # 便于后端生成回调/重定向绝对路径
}
# 后台管理 Web 静态产物
location /koala/ {
    alias /opt/koala/www/;
    try_files $uri $uri/ /koala/index.html;       # SPA history 路由兜底
}
# TLS 终止 + HSTS
```

- **前端 baseURL**：C 端小程序与后台 Web 均配 `https://域名/koala/api/v1/`；后台静态资源 publicPath/base 配 `/koala/`。
- **支付回调地址**：微信 notify_url 配成带前缀的对外完整 URL `https://域名/koala/api/v1/order/pay-notify`（Nginx 剥前缀后落到后端 `/api/v1/order/pay-notify`）。
- **Actuator**：仍走后端原生 `/actuator/**`（限内网/容器网络访问，不经 `/koala` 公网暴露），Compose healthcheck 路径不变。

C 端小程序经微信审核后访问 `https://域名/koala/api/v1/`。

### 11.4 三端构建产物
**构建一律在本地完成**（2C2G 服务器不编译，避免 OOM）：

| 端 | 技术 | 构建命令 | 产物 | 部署去向 |
|----|------|---------|------|---------|
| 后台管理 | Vue3 + Element Plus + Vite | `pnpm build` | `dist/` 静态文件 | rsync/scp 到服务器 Nginx 目录(`/koala/`) |
| C 端 | uni-app(Vue3) | `pnpm build:mp-weixin` | `dist/mp-weixin/` | 微信开发者工具上传审核 |
| 后端 | Spring Boot + Maven | `mvn -B clean package -DskipTests` → `docker build` | Docker 镜像 `koala/server:<tag>` | `docker save` 打包 → ssh 传服务器 `docker load` |

> 后端在本地编译打镜像，经 `docker save | ssh … docker load` 传到服务器加载运行（不经镜像仓库）；后台 Web 静态产物本地构建后同步到服务器 Nginx 目录；C 端小程序产物因需走微信审核，由开发者工具人工上传。镜像 tag 用 git commit 短 hash，便于回滚精确切版本。

### 11.5 手动部署（本地构建 → 推送服务器，执行脚本）

代码托管 GitHub。**2C2G 服务器不编译**——构建在本地完成（编译吃内存），把镜像与静态产物传到服务器再运行，**单机个人项目，无需 CI/CD，一条命令搞定**。

**首次准备**（仅一次）：
- 本地装好 Docker + Node/pnpm，能 SSH 免密登录服务器。
- 服务器装好 Docker + Docker Compose；克隆仓库到 `/opt/koala` 仅为放 `docker-compose.yml`、`nginx.conf`、`.env`（DB/Redis/微信密钥/JWT 密钥等敏感项，不进仓库）；服务器**不需要 JDK/Maven/pnpm**。

**日常发布**：在**本地**项目目录执行 `./deploy.sh`，脚本自动完成「本地构建 → 传输 → 远程运行」：

```bash
#!/usr/bin/env bash
# deploy.sh —— 本地执行:本地构建+打镜像 → 传服务器 → 远程加载运行,幂等可重复
set -euo pipefail
HOST="user@阿里云IP"; REMOTE=/opt/koala
TAG=$(git rev-parse --short HEAD)             # 镜像版本=当前 commit

# ── 本地构建 ──
# 1. 后端:编译打镜像(本地吃内存,与服务器无关)
mvn -B -f server/pom.xml clean package -DskipTests
docker build -t koala/server:$TAG ./server

# 2. 后台 Web 静态产物
pnpm -C dashboard install --frozen-lockfile
pnpm -C dashboard build

# ── 传输到服务器 ──
docker save koala/server:$TAG | gzip | ssh "$HOST" 'gunzip | docker load'   # 3. 镜像直传(不经仓库)
rsync -az --delete dashboard/dist/ "$HOST:$REMOTE/www/"                      # 4. 静态产物同步

# ── 远程运行 ──
ssh "$HOST" "cd $REMOTE \
  && sed -i 's/^SERVER_TAG=.*/SERVER_TAG=$TAG/' .env \
  && docker compose up -d --no-deps server \
  && docker image prune -f"                   # 健康检查通过再切流量;清理旧镜像

echo "✅ 部署完成 (tag=$TAG)"
```

> 看日志：`ssh "$HOST" 'cd /opt/koala && docker compose logs -f server'`；DB 迁移由 **Flyway** 在 server 启动时自动执行。
> 镜像 ~200-400M 走家用上行带宽传输，通常一两分钟；嫌慢或发布频繁可改用 ACR 内网拉取（compose 与远程 `up` 几乎不变，仅把 `save|load` 换成 `push`/`pull`）。
> C 端小程序产物单独用微信开发者工具构建上传审核，不在此脚本内。

### 11.6 回滚（执行脚本）

后端镜像按 commit tag 保留在服务器上（`docker image prune -f` 只清无 tag 的悬空镜像，**带 tag 的历史版本不会被删**），回滚直接**切回旧镜像 tag**，无需重新编译，秒级生效。在**本地**执行 `./rollback.sh [commit]`（不传参默认回上一个 commit）：

```bash
#!/usr/bin/env bash
# rollback.sh [target_commit] —— 本地执行:切回旧镜像 tag,后端秒级回滚;静态产物从该 commit 重建
set -euo pipefail
HOST="user@阿里云IP"; REMOTE=/opt/koala
TARGET=$(git rev-parse --short "${1:-HEAD~1}")   # 默认回上一个 commit
echo "↩ 回滚到 $TARGET"

# 1. 后端:服务器若已有该 tag 镜像则直接切;没有(已被清理)则本地重建并传上去
if ! ssh "$HOST" "docker image inspect koala/server:$TARGET >/dev/null 2>&1"; then
  echo "服务器无 $TARGET 镜像,本地重建..."
  git stash; git checkout "$TARGET"
  mvn -B -f server/pom.xml clean package -DskipTests
  docker build -t koala/server:$TARGET ./server
  docker save koala/server:$TARGET | gzip | ssh "$HOST" 'gunzip | docker load'
  git checkout -; git stash pop || true
fi

# 2. 后台静态产物:从目标 commit 重建并同步(静态产物不带版本,必须重建)
git stash; git checkout "$TARGET"
pnpm -C dashboard install --frozen-lockfile && pnpm -C dashboard build
rsync -az --delete dashboard/dist/ "$HOST:$REMOTE/www/"
git checkout -; git stash pop || true

# 3. 远程切 tag 重启
ssh "$HOST" "cd $REMOTE \
  && sed -i 's/^SERVER_TAG=.*/SERVER_TAG=$TARGET/' .env \
  && docker compose up -d --no-deps server"

echo "✅ 回滚完成 (tag=$TARGET)"
```

> DB 迁移由 **Flyway** 只增量、向后兼容（先兼容后清理），切回旧镜像不会破坏已迁移的 schema；如需回退表结构，需另写向前兼容的增量迁移，**禁手改、禁回退已执行的迁移脚本**。
> 服务器镜像不会无限堆积：保留近几个 commit tag 即可，更早的可手动 `docker rmi koala/server:<旧tag>` 清理。

### 11.7 备份与容灾
- MySQL 每日 mysqldump→gzip→对象存储，保留 30 天；定期演练恢复。
- Redis AOF 持久化；关键业务态（订单/券/库存）以 MySQL 为准，Redis 仅加速。

### 11.8 扩展演进路径
- 单机 → 数据库主从读写分离 → 应用多实例 + LB（应用无状态，加机器即可横向扩展）。
- 易变点（支付/存储/物流/登录）已用 SPI 接口隔离，换渠道/供应商不改调用方。
