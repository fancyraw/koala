# 整体技术设计

> 本文档基于《C端产品设计.md》《后台管理产品设计.md》对齐最新需求，给出**可在正式环境部署、通用可扩展**的工程方案。覆盖：分层架构与扩展点、领域与数据模型、核心算价/下单/支付/售后/券机制、幂等并发、安全、可观测性与部署运维。
>
> 业务基线：**无会员体系、无积分体系**——无会员等级、无升级送券、无下单折扣、无积分。**全部商品统一参与满减券与无门槛券，无单品营销开关**；优惠券由系统统一向全员自动下发，无人群定向 / 标签机制，每模版每人 1 张。

---

## 一、设计目标与原则

### 1.1 目标
- **生产可部署**：多环境隔离、密钥外置、HTTPS、健康检查、可观测、可备份恢复、可灰度回滚。
- **通用可扩展**：核心能力以接口抽象（支付/存储/物流/登录/通知），新增渠道不改业务代码；业务策略走配置中心，改参数不发版。
- **数据一致**：资金与库存/券资产强一致，幂等防重，事务边界清晰。

### 1.2 架构原则
| 原则 | 落地方式 |
|------|---------|
| 分层清晰 | Controller → Service(应用) → Domain(领域) → Mapper(持久层)，依赖单向向下 |
| 面向扩展 | 易变点定义 SPI 接口 + 多实现，由配置选择激活实现（策略模式 / 工厂） |
| 配置外置 | 环境差异走 Spring Profile + 环境变量；业务策略走 `sys_config`（运行期可调） |
| 事件解耦 | 订单支付成功等关键节点发领域事件，副作用（销量统计、日志、预留通知）由监听器处理 |
| 约定优于配置 | 统一响应体、统一错误码、统一分页、统一鉴权与日志切面 |
| 安全默认 | 默认最小权限、默认脱敏、默认参数校验、默认幂等 |

## 二、整体架构

### 2.1 系统组成

```
                          用户微信小程序        运营/客服浏览器
                                │                    │
                          ┌─────▼────────────────────▼─────┐
                          │        Nginx 反向代理            │  :443 (TLS)
                          │  /            → 后台管理 Web 静态  │
                          │  /api/v1/     → 后端服务          │
                          │  /api/v1/admin→ 后端服务(后台)    │
                          └─────────────┬───────────────────┘
                                        │
                              ┌─────────▼─────────┐
                              │   后端服务 Spring   │  :8080 (可水平扩展 N 实例)
                              │  Controller/Service │
                              │  Domain/Mapper      │
                              │  + Actuator 健康检查 │
                              └──┬──────┬──────┬────┘
                  ┌──────────────┘      │      └──────────────┐
            ┌─────▼─────┐         ┌──────▼──────┐       ┌──────▼───────┐
            │   MySQL    │         │    Redis     │       │  外部服务抽象  │
            │ 主(+可选从) │         │ 缓存/锁/幂等  │       │ 支付/存储/物流 │
            └───────────┘         └─────────────┘       └──────────────┘
```

- **无状态后端**：会话用 JWT，服务器不存会话态，后端可多实例水平扩展（Nginx 轮询 / 最少连接）。
- **共享态下沉 Redis**：分布式锁、幂等令牌、限流计数、热点缓存全部放 Redis，保证多实例一致。

### 2.2 技术栈

| 层级 | 选型 | 说明 |
|------|------|------|
| C 端 | uni-app（Vue3） | 编译为微信小程序 |
| 后台管理 | Vue3 + Element Plus + Vite | SPA，Nginx 托管静态产物 |
| 后端 | Spring Boot 2.7 + JDK 8 | RESTful；MyBatis-Plus 持久层 |
| 数据库 | MySQL 5.7（InnoDB / utf8mb4） | 业务主存储；Flyway 管理 schema 迁移 |
| 缓存 | Redis 7 | 会话黑名单、幂等、分布式锁(Redisson)、限流、热数据 |
| 连接池 | HikariCP | 默认连接池 |
| 文件存储 | 对象存储抽象（七牛 / 阿里 OSS / S3 任选） | 默认七牛，接口隔离可切换 |
| 支付 | 支付渠道抽象（默认微信 JSAPI） | `PaymentChannel` 接口，可扩展支付宝等 |
| 物流 | 物流查询抽象（默认快递100） | `LogisticsProvider` 接口，可扩展快递鸟 |
| 登录 | 登录方式抽象（C 端微信、后台扫码/手机号） | `user_auth` 多端凭证表支撑 |
| 可观测 | Spring Actuator + 文件日志（带 traceId）+ operation_log 表 | 个人开发者轻量方案，不引 Prometheus/Grafana/ELK |
| 部署 | Docker + Docker Compose（阿里云单机） | 手动 `git pull` + 执行部署脚本，无需 CI/CD |

### 2.3 项目结构

```
koala/
├── server/                     # Spring Boot 后端
│   ├── src/main/java/com/koala/
│   │   ├── common/             # 统一响应/异常/错误码/工具/切面(鉴权·日志·限流)
│   │   ├── config/             # Spring 配置(Security/Redis/MyBatis/Swagger)
│   │   ├── infra/              # 基础设施：扩展点 SPI 接口 + 实现
│   │   │   ├── payment/        #   PaymentChannel + WechatPayChannel
│   │   │   ├── storage/        #   FileStorage + QiniuStorage
│   │   │   ├── logistics/      #   LogisticsProvider + Express100Provider
│   │   │   └── notify/         #   预留：消息/通知抽象
│   │   ├── modules/            # 业务模块(按领域纵向切分,高内聚)
│   │   │   ├── user/  product/  coupon/  cart/
│   │   │   ├── order/ aftersale/ logistics/
│   │   │   └── admin/ content/  config/  ops/
│   │   └── KoalaApplication.java
│   └── src/main/resources/
│       ├── application.yml             # 公共配置
│       ├── application-{dev,test,prod}.yml
│       └── db/migration/               # Flyway 版本化 SQL

├── web/                        # C 端 uni-app
├── deploy/                     # docker-compose、Nginx、env 模板

> **模块化原则**：业务按领域纵向切分（user/product/order…），每个模块内自带 controller/service/mapper，跨模块只经 service 接口通信、不直接跨模块查表，便于未来拆分微服务。

## 三、扩展性设计（核心：易变点抽象）

把「会随渠道/供应商变化」的能力定义为接口，业务代码只依赖接口；具体实现由配置选择，新增渠道**不改调用方**。

### 3.1 支付渠道抽象

```java
public interface PaymentChannel {
    String code();                                   // wechat / alipay ...
    PrepayResult prepay(PrepayCommand cmd);          // 统一下单，返回前端调起参数
    NotifyResult parseNotify(HttpServletRequest r);  // 解析+验签回调
    RefundResult refund(RefundCommand cmd);          // 退款
}
// WechatPayChannel 实现微信 JSAPI；扩展支付宝只需新增 AlipayChannel 并注册
// 通过 PaymentChannelFactory.get(channelCode) 路由，channelCode 存于 payment 表
```

### 3.2 文件存储抽象

```java
public interface FileStorage {
    UploadCredential issueCredential(String bizType);  // 前端直传凭证
    String publicUrl(String key);                      // 拼访问 URL
    void delete(String key);
}
// 默认 QiniuStorage；切换 OSS/S3 改配置 koala.storage.provider 即可
```

### 3.3 物流查询抽象

```java
public interface LogisticsProvider {
    List<TrackNode> query(String company, String trackingNo);  // 轨迹节点(最新在前)
}
// 默认 Express100Provider；结果带缓存层，详见 4.8
```

### 3.4 业务策略走 sys_config
运费 / 包邮门槛、支付商户号、订单超时分钟数等**易调策略**统一进 `sys_config` 表，应用启动加载到本地缓存，后台/接口变更后失效刷新（见 7.3）。**改策略不发版**。

### 3.5 领域事件（解耦副作用，便于增量扩展）

```
OrderPaidEvent  → [销量+1] [业务日志] [预留:推送通知]
RefundedEvent   → [券原路退回] [业务日志]
```
新增副作用只加监听器，不动主流程。

## 四、领域模型

```
┌─────────┐    ┌─────────┐      ┌─────────┐ 1:N ┌─────────┐
│  用户     │1:N │  地址     │      │  分类     │────▶│  商品     │ 1:N ┌────────┐

│         │    └─────────┘      └─────────┘     └────┬────┘     └────────┘

└────┬─────┘                                          │

     ├───────▶│ 用户优惠券  │────▶┌─────────┐ │      购物车 CartItem  │
     │        │UserCoupon │     │ 券模板    │ └─────────────────────┘

              └──────────────────────────────────────────┘

```

## 五、数据库设计

> 规范：引擎 InnoDB、字符集 utf8mb4；金额 `DECIMAL(10,2)`；时间统一 `created_at/updated_at`；并发更新的表加乐观锁 `version`；后台维护的表加 `created_by/updated_by` 审计字段。Schema 由 **Flyway** 版本化管理，正式环境**只增量、禁手改**。

### 5.1 用户与登录

```sql
-- 用户表（无会员体系、无折扣、无积分）
CREATE TABLE user (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    nickname        VARCHAR(64)  DEFAULT '' COMMENT '昵称',
    avatar_url      VARCHAR(512) DEFAULT '' COMMENT '头像URL',
    phone           VARCHAR(20)  DEFAULT '' COMMENT '手机号',
    status          TINYINT      DEFAULT 1 COMMENT '1=正常 0=禁用',
    created_at      DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_phone (phone)
) COMMENT='用户表';
-- 无会员等级 / 累计消费 / 折扣 / 积分。优惠券通投全员、无新人/标签机制

-- 用户登录凭证表（多端可扩展：当前 wechat_mp，未来 phone/apple/…）
CREATE TABLE user_auth (
    id         BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id    BIGINT NOT NULL,
    auth_type  VARCHAR(16)  NOT NULL COMMENT 'wechat_mp/phone/email',
    auth_id    VARCHAR(128) NOT NULL COMMENT 'openid/手机号/邮箱',
    union_id   VARCHAR(64)  DEFAULT '' COMMENT '微信 unionid(跨应用打通预留)',
    created_at DATETIME     DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_type_id (auth_type, auth_id),
    INDEX idx_user_id (user_id)
) COMMENT='用户登录凭证表';

-- 用户地址表
CREATE TABLE user_address (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    name VARCHAR(32) NOT NULL, phone VARCHAR(20) NOT NULL,
    province VARCHAR(32) NOT NULL, city VARCHAR(32) NOT NULL, district VARCHAR(32) NOT NULL,
    detail VARCHAR(256) NOT NULL, full_address VARCHAR(512) NOT NULL,
    is_default TINYINT DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_user_id (user_id)
) COMMENT='用户地址表';
```

### 5.2 商品与分类

```sql
CREATE TABLE category (

    status TINYINT DEFAULT 1 COMMENT '1=上架 0=下架',

-- 商品规格 SKU（售价唯一来源；库存乐观锁防超卖）
CREATE TABLE product_sku (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    product_id BIGINT NOT NULL,
    name VARCHAR(64) NOT NULL COMMENT '规格名',
    price DECIMAL(10,2) NOT NULL COMMENT '售价',
    stock INT DEFAULT 0 COMMENT '库存',
    sort_order INT DEFAULT 0,
    version INT DEFAULT 0 COMMENT '乐观锁(扣减库存)',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_product_id (product_id)
) COMMENT='商品规格表(最多4个/商品);库存<20 C端售罄预警阈值在业务层';
```

### 5.3 优惠券（券模板 + 用户券）

```sql
-- 券模板（规则定义）：后台只管模板，用户券由系统下发生成
CREATE TABLE coupon (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(64) NOT NULL,
    type TINYINT NOT NULL COMMENT '1=满减券 2=无门槛券',
    discount_amount DECIMAL(10,2) NOT NULL COMMENT '优惠金额',
    min_spend DECIMAL(10,2) DEFAULT 0.00 COMMENT '满减门槛(无门槛=0)',
    total_count INT NOT NULL COMMENT '发行总量',
    issued_count INT DEFAULT 0 COMMENT '已发(下发的用户券数)',
    used_count INT DEFAULT 0 COMMENT '已用',
    valid_type TINYINT DEFAULT 1 COMMENT '1=固定区间 2=领取后N天',
    start_time DATETIME DEFAULT NULL, end_time DATETIME DEFAULT NULL,
    valid_days INT DEFAULT 0 COMMENT 'valid_type=2 时领取后有效天数',
    status TINYINT DEFAULT 1 COMMENT '1=正常 0=已停发',
    version INT DEFAULT 0 COMMENT '乐观锁(防超发)',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) COMMENT='券模板表(已开始即有下发记录的不可删除,只能停发)';

-- 用户券（实例，含锁定态支撑资产冻结释放）
CREATE TABLE user_coupon (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    coupon_id BIGINT NOT NULL,
    status TINYINT DEFAULT 0 COMMENT '0=未使用 1=已使用 2=已过期 3=已锁定(下单未支付)',
    lock_order_no VARCHAR(32) DEFAULT NULL COMMENT '锁定/使用的订单号',
    expire_at DATETIME NOT NULL COMMENT '到期时间(冗余,临期倒计时与过期任务)',
    granted_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '下发时间',
    used_at DATETIME DEFAULT NULL,
    UNIQUE KEY uk_user_coupon (user_id, coupon_id) COMMENT '每模版每人至多1张,兜底防并发重复下发',
    INDEX idx_user_status (user_id, status),
    INDEX idx_expire (status, expire_at)
) COMMENT='用户优惠券表';
```

### 5.4 订单 / 支付 / 售后

```sql
-- 订单主表（金额口径与产品算价一致；user_deleted 实现C端软删除）
CREATE TABLE `order` (

    user_deleted TINYINT DEFAULT 0 COMMENT 'C端删除(软删,后台仍可见)',

CREATE TABLE order_item (

## 六、核心模块设计

### 6.1 登录与鉴权

```
C端：wx.login() 取 code → /api/v1/user/login
   → code 换 openid → 查 user_auth(wechat_mp, openid)
     命中=登录；未命中=单事务创建 user + user_auth(自动注册)
   → 签发 JWT(含 userId、端类型)；触发自动领券查询(通投全员、每模版每人1张,见 6.6)
后台：扫码(主)/手机号验证码(备) → 匹配 admin → 签发 JWT(含 adminId)
鉴权：网关切面校验 JWT；后台为单一管理员角色，登录即可访问全部模块(不分角色)
JWT：短期 access + Redis 维护登出黑名单；禁用用户即时失效(status=0 校验)
```

### 6.2 价格计算（与产品口径强一致）

**固定优先级逐级扣减，金额均不为负**。所有商品统一参与满减/无门槛券，**无任何折扣**：

```
1. 商品合计 = Σ(sku.price × qty)
2. 券抵扣(无门槛 + 满减 各一张叠加)
   - 满减券：门槛以「商品合计」判断并按面额直减
   - 无门槛券：直接按面额直减
   - 券后小计 = 商品合计 − (无门槛面额 + 满减面额)，下限 0
3. 运费：满额包邮门槛以「商品合计」判断；运费不参与券抵扣
4. 实付 = 商品合计 − 券抵扣 + 运费（微信支付要求最低 0.01）
```

- `/api/v1/order/preview` 与 `/submit` **共用同一算价服务**，保证预览=下单。
- **券组合默认始终最优**：系统在用户可用券中自动选「无门槛 + 满减各一张」抵扣最大的组合，无开关。
- 商详页只返回售价，不算「到手价」；券优惠统一在购物车/确认页体现。

### 6.3 下单 + 资产冻结（先锁定再支付，防货财两空）

```
提交订单（单事务 + 用户维度分布式锁防并发）：
  a. 幂等：前端 submitToken 经 Redis SETNX，重复提交拦截
  b. 扣库存：乐观锁 UPDATE stock=stock-N WHERE id=? AND stock>=N（影响0行=库存不足）
  c. 算价(6.2)，落 order 各金额字段
  d. 锁定券：user_coupon 0→3(锁定)、lock_order_no=订单号；写 order_coupon
  e. 建 order(status=0) + order_item
  f. expire_at = now + 超时分钟；返回订单号
  → 调支付渠道 prepay()，前端 wx.requestPayment()
```

### 6.4 支付结果处理（释放/核销，最高优先级）

```
支付成功回调 /api/v1/order/pay-notify（验签 → 幂等：transaction_id 唯一 + 状态 CAS）：
  - order 0→1(待发货)（CAS，已处理直接返回成功）
  - user_coupon 3→1(已使用)、coupon.used_count+1
  - payment 置成功
  - 发 OrderPaidEvent → 监听器：销量+1、业务日志

支付失败/取消/超时(定时任务)→ 释放资产（单事务）：
  - 回滚库存 stock+=N；释放券 user_coupon 3→0；order→4(已取消)
⚠ 资产释放必须与失败/取消同事务、最高优先级，杜绝「券没了」投诉
```

### 6.5 售后 / 退款

```
状态机：0待审核 →1通过待寄回 →(退货退款)2买家已寄回 →3商家已收货 →4已退款
                                └(仅退款,未发货)──────────────────→4已退款
                     └5已拒绝
仅退款(未发货)：商家审核通过 → 直接原路退款 → 4
退货退款(已发货)：审核通过 → 买家寄回(填单号) → 商家确认收货 → 退款 → 4
退款处理(事务)：调渠道 refund()→payment=3；未过期券原路退回 user_coupon→0；
              发 RefundedEvent → 按策略回滚销量
```

### 6.5.1 券模版状态流转（持久态 + 时间派生）

> **数据库只存 2 个持久态**（`coupon.status`：`1=正常` / `0=已停发`）。UI 上的「未开始 / 进行中 / 已结束」**不落库**，而是 `status=1` 时由 `start_time / end_time` 与当前时间**实时派生**——查询时按 `now` 计算，无需定时任务推进。

```
状态派生（status=1 正常时，按 now 判定）：
  now < start_time            → 未开始（灰）
  start_time ≤ now ≤ end_time → 进行中（绿）  ◀── 唯一可下发窗口
  now > end_time              → 已结束（灰）

人工流转（唯一写操作）：
  任意时刻点「停发」→ status=0 已停发（灰，终态·不可逆）

优先级：status=0 盖过时间派生——即使仍在有效期内，停发后即显示「已停发」并立即停止下发
```

- **未开始 → 进行中 → 已结束**：时间自然推进，无人工操作、无定时任务，查询实时判定
- **停发**：唯一人工流转，置 `status=0`，**终态不可逆**；停发后不再生成新 user_coupon，**已发出的用户券不受影响**（各自用到过期）
- **仅「进行中」可下发**：[6.6](#) 候选券判定 `status=1 且在有效期内` 即等价于「进行中」；未开始 / 已结束 / 已停发均不下发
- **删除约束**：仅「未开始且无任何下发记录（issued_count=0）」可删；一旦有下发记录只能停发（见 `coupon` 表注释）

### 6.6 统一领券弹窗（系统下发，无手动领取 = 后端自动领）

> **「无手动领取」的技术实质 = 后端自动领。** 前端不调「领取」动作，登录/进首页时调一次「查询可领券」接口，后端把所有「进行中」且该用户尚未领过的券模板各发 1 张并**自动生成 user_coupon**，再把本次新到账的券返回给前端弹窗告知。用户只感知「已为你发放 N 张」。**通投全员、无标签 / 资格判定、每模版每人 1 张。**

```
登录/进首页 → /api/v1/coupons/auto-grant（查询并自动领，幂等）：
  1. 候选券模板：status=1正常、在有效期内、issued_count<total_count
  2. 去重过滤：该用户对此券未领过（user_coupon 不存在 user_id+coupon_id 记录；
     领过即不再发，含已用/已过期，无需算资格、无标签）
  3. 对通过的券：用户维度分布式锁 + 单事务下发
     - 用户级幂等键(user:{id}:coupon:{couponId}) 防并发重复领
       (配合 user_coupon 唯一索引 uk(user_id,coupon_id) 兜底)
     - coupon.version 乐观锁 + issued_count<total_count 条件更新(防超发)
     - 建 user_coupon(status=0)、coupon.issued_count+1
     - expire_at 按 valid_type 计算(固定区间 or 领取后N天)
  4. 返回本次新到账券列表 → 前端弹「领券弹窗」，按钮「去使用」
异常兜底：
  - 接口失败本次不弹、下次再查(前端不阻塞浏览)
  - 部分下发失败：已成功的正常返回，失败的下次重试
```

> **券一经下发即为用户资产。** user_coupon 下发后归用户所有，券模板停发 / 结束**不影响已到账券的使用**，与「已发出的用户券不受影响、杜绝券没了投诉」一致。

### 6.7 物流查询（带缓存）

```
发货：后台填 logistics_company + logistics_no → shipped_at、status→2
查看物流：/orders/{no}/logistics → LogisticsProvider.query()
  → 结果缓存 Redis(~30min)，降低第三方调用与费用
  → 节点最新在前；C端弹窗节点>5 默认 3-4 条 + 全屏物流页；待发货无入口
```

### 6.8 图片上传
后台经 `FileStorage.issueCredential()` 取直传凭证 → 前端直传对象存储 → 回存 CDN URL。**不经后端中转**，省带宽。

## 七、关键技术机制

### 7.1 幂等与防重

| 场景 | 方案 |
|------|------|
| 提交订单 | 前端 submitToken(一次性) + Redis SETNX；按钮点击即置灰 loading |
| 支付回调 | transaction_id 唯一键 + 订单状态 CAS，重复回调返回成功 |
| 确认收货 | 状态机 CAS(status=2→3，影响0行视为已处理) + 前端置灰 |
| 自动领券 | 用户级幂等键 + user_coupon 唯一索引(每模版每人1张) + 发行量乐观锁；通投全员、无资格判定 |
| 售后申请 | order_no 维度当前状态校验，重复申请拦截 |
| 通用写 | 优先数据库唯一约束/乐观锁，不依赖前端 |

### 7.2 并发与一致性
- **库存防超卖**：乐观锁 `UPDATE … WHERE stock>=N`，无行锁竞争、无超卖。
- **券防超发**：`coupon.version` 乐观锁 + `issued_count<total_count` 条件更新。
- **分布式锁**：Redisson 按 `user:{id}` 锁下单关键段，多实例下仍互斥。
- **事务边界**：资金/库存/券变更同事务；第三方调用（支付/退款）放事务外或用「本地消息+对账」避免长事务。

### 7.3 缓存策略
| 数据 | 策略 |
|------|------|
| 首页聚合/分类/Banner | Redis 缓存 + 后台变更主动失效 |
| sys_config 业务策略 | 启动加载本地缓存，变更发失效信号刷新 |
| 物流轨迹 | 短 TTL（30min） |
| 防击穿 | 热点 key 互斥重建；空值短缓存防穿透 |

### 7.4 定时任务
| 任务 | 频率 | 说明 |
|------|------|------|
| 订单超时取消 | 每 1 分钟 | status=0 且 expire_at<now → 释放库存/券 |
| 优惠券过期 | 每天凌晨 | user_coupon 到期 → status=2 |
| 监控指标快照 | 每 1 分钟 | 拉 Actuator metrics + Redis 限流计数 → 写 metrics_snapshot（支撑系统监控趋势曲线） |
| 监控数据清理 | 每天凌晨 | 删除 metrics_snapshot 超过保留期（如 N 天）的旧快照 |
| 数据库备份 | 每天 3:00 | mysqldump→gzip→对象存储，留 30 天 |

> 多实例下定时任务需防重复执行：用分布式锁或 ShedLock 保证集群内单次执行。

## 九、接口规范

### 9.1 通用约定
- **路径前缀**：`/api/v1/`，版本化便于不破坏式升级；后台 `/api/v1/admin/`。
- **统一响应体**：
  ```json
  { "code": 0, "message": "ok", "data": {}, "traceId": "..." }
  ```
  `code=0` 成功；非 0 为业务错误码（分段：1xxx 鉴权、2xxx 参数、3xxx 业务、5xxx 系统）。
- **分页**：`?page=1&size=20`，返回 `{ list, total, page, size }`。
- **错误处理**：全局异常处理器统一兜底，区分业务异常（可展示文案）与系统异常（脱敏 + 告警）。
- **校验**：入参 Bean Validation；金额/状态等服务端二次校验，不信前端。

> **方法约定**：全站只用 **GET（查询）** 与 **POST（写操作）**，不用 PUT/DELETE。更新走 `/update`、删除走 `/delete` 等动词化路径，个人项目更省心、网关/代理兼容性也更好。

## 十、可观测性（个人开发者轻量方案）

> 不引 Prometheus / Grafana / ELK 等独立基建——单机个人项目用 **Actuator + 文件日志 + 数据库表**就够，运维数据直接由后台「运维管理」页查接口/查表展示。

| 维度 | 方案 | 对应后台 |
|------|------|---------|
| 健康检查 | Spring Actuator `/actuator/health`（DB/Redis/diskSpace 探针）+ `/actuator/metrics`（JVM/CPU/内存），Nginx/容器探活 | 运维-系统健康 |
| 系统监控 | Actuator `http.server.requests`(QPS/p95) + HikariCP/Tomcat/GC metrics + `logback.events`(error) + Redis 限流计数；定时任务每分钟快照入 `metrics_snapshot` 表以出趋势曲线 | 运维-系统监控 |
| 日志 | Logback 文件日志 + MDC traceId 全链路串联，按天滚动 + 大小切分保留 N 天 | 服务器 `logs/` 目录 |
| 业务/审计日志 | operation_log：operator_type=1 C端业务、=2 管理员审计、=3 系统；带 traceId 关联文件日志 | 运维-操作日志(单列表,按主体筛选) |
| 告警 | 轻量：关键错误/支付失败在监听器里直接发邮件/微信通知（无独立告警系统） | — |

> **系统健康零额外成本**：服务 / DB / Redis / 磁盘状态与 JVM / CPU / 内存指标全部取自 Actuator `health` / `metrics` 端点，引入 `spring-boot-starter-actuator` 后开箱即有，**后端无任何额外埋点**，后台「系统健康」页仅做接口拉取 + 展示。Docker 单机部署下 Actuator 看的是容器内视角（堆内存、容器挂载磁盘），足够回答「服务活着、堆没爆、盘没满」；**不监控宿主机级指标**，避免引额外 agent。`/actuator/**` 端点须走管理员鉴权或限内网/容器网络访问，不裸暴露公网。

## 十一、部署与运维

### 11.1 多环境
`dev / test / prod` 三套 Spring Profile；差异（DB、Redis、密钥、域名）全走**环境变量 / Docker Secret**，镜像一份、配置分离，支持同镜像多环境部署。

### 11.2 Docker Compose（单机起步）

```yaml
services:
  mysql:  { image: mysql:5.7, env_file: .env, volumes: ["./data/mysql:/var/lib/mysql"] }
  redis:  { image: redis:7-alpine, command: ["redis-server","--appendonly","yes"], volumes:["./data/redis:/data"] }
  server:
    build: ./server
    env_file: .env                 # DB/Redis/微信密钥/JWT密钥 等敏感项
    depends_on: { mysql: {condition: service_healthy}, redis: {condition: service_started} }
    healthcheck: { test: ["CMD","curl","-f","http://localhost:8080/actuator/health"], interval: 30s }
    deploy: { replicas: 2 }        # 无状态，可多实例
  nginx:
    image: nginx:alpine
    ports: ["80:80","443:443"]
    volumes: ["./deploy/nginx.conf:/etc/nginx/nginx.conf:ro","./deploy/certs:/etc/nginx/certs:ro"]
    depends_on: [server]
```

> 数据库迁移由 **Flyway** 在 server 启动时自动执行，保证 schema 与代码版本一致。

### 11.3 Nginx 路由
```
/api/v1/   → 后端服务(:8080，多实例 upstream 负载)
/          → 后台管理 Web 静态产物
TLS 终止 + HSTS；/api/ 开启限流(limit_req) 兜底
```
C 端小程序经微信审核后访问 `https://域名/api/v1/`。

### 11.4 三端构建产物
三端构建命令统一，本地或服务器上执行：

| 端 | 技术 | 构建命令 | 产物 | 部署去向 |
|----|------|---------|------|---------|
| 后台管理 | Vue3 + Element Plus + Vite | `pnpm build` | `dist/` 静态文件 | Nginx 静态目录(`/`) |
| C 端 | uni-app(Vue3) | `pnpm build:mp-weixin` | `dist/mp-weixin/` | 微信开发者工具上传审核 |
| 后端 | Spring Boot + Maven | `mvn -B clean package -DskipTests` | 可执行 jar | Docker 镜像 |

> 后端打镜像在服务器本地构建运行；后台 Web 静态产物同步到服务器 Nginx 目录；C 端小程序产物因需走微信审核，由开发者工具人工上传。

### 11.5 手动部署（阿里云服务器，执行脚本）

代码托管 GitHub，部署时在阿里云服务器上 `git pull` 拉最新代码，再执行部署脚本——**单机个人项目，无需 CI/CD，一条命令搞定**。

**首次准备**（仅一次）：服务器装好 Docker + Docker Compose，克隆仓库到 `/opt/koala`，准备好 `.env`（DB/Redis/微信密钥/JWT 密钥等敏感项，不进仓库）。

**日常发布**：SSH 登录服务器，进入项目目录执行 `./deploy.sh`：

```bash
#!/usr/bin/env bash
# /opt/koala/deploy.sh —— 手动部署脚本，幂等、可重复执行
set -euo pipefail
cd /opt/koala

git pull origin main                          # 1. 拉最新代码

# 2. 构建后台管理 Web 静态产物 → Nginx 目录
pnpm -C dashboard install --frozen-lockfile
pnpm -C dashboard build
rm -rf /opt/koala/www && cp -r dashboard/dist /opt/koala/www

# 3. 构建后端镜像并重启（Compose 内 build server）
docker compose build server
docker compose up -d --no-deps server         # 健康检查通过再切流量
docker image prune -f                          # 清理旧镜像

echo "✅ 部署完成"
```

> 执行前后看日志：`docker compose logs -f server`；DB 迁移由 **Flyway** 在 server 启动时自动执行。
> C 端小程序产物单独用微信开发者工具构建上传审核，不在此脚本内。

### 11.6 回滚（执行脚本）

发布后若出问题，SSH 登录服务器执行 `./rollback.sh [commit]`（不传参默认回滚到上一个 commit）：

```bash
#!/usr/bin/env bash
# /opt/koala/rollback.sh [target_commit] —— 回滚脚本
set -euo pipefail
cd /opt/koala

TARGET="${1:-HEAD~1}"                          # 默认回上一个 commit
echo "↩ 回滚到 $TARGET"
git reset --hard "$TARGET"                     # 1. 代码回退到指定版本

# 2. 重新构建后台静态产物 → Nginx 目录
pnpm -C dashboard install --frozen-lockfile
pnpm -C dashboard build
rm -rf /opt/koala/www && cp -r dashboard/dist /opt/koala/www

# 3. 重建后端镜像并重启
docker compose build server
docker compose up -d --no-deps server
docker image prune -f

echo "✅ 回滚完成（当前 $(git rev-parse --short HEAD)）"
```

> DB 迁移由 **Flyway** 只增量、向后兼容（先兼容后清理），代码回退不会破坏已迁移的 schema；如需回退表结构，需另写向前兼容的增量迁移，**禁手改、禁回退已执行的迁移脚本**。

### 11.7 备份与容灾
- MySQL 每日 mysqldump→gzip→对象存储，保留 30 天；定期演练恢复。
- Redis AOF 持久化；关键业务态（订单/券/库存）以 MySQL 为准，Redis 仅加速。

### 11.8 扩展演进路径
- 单机 → 数据库主从读写分离 → 应用多实例 + LB → 热点模块（如订单/库存）拆服务。
- 当前模块化纵向切分 + 接口隔离，已为后续服务化预留边界，无需推倒重来。
